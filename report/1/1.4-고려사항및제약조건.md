### 1.4. 고려사항 및 제약조건

본 절은 설계 이전에 외부에서 이미 확정된 사실과 반드시 준수해야 하는 제약조건을 정리한다. 이 제약들이 아키텍처 결정에 어떻게 작용하는지(각 제약이 어떤 설계 선택지를 배제하는지)는 2.3 아키텍처 제약사항에서 별도로 정리한다.

#### 제약조건 요약

| ID | 제약 | 핵심 내용 |
| :---: | ----- | ----- |
| C-01 | 현행 기술 스택 고정 | Java Spring Boot + HikariCP + Feign(Hystrix CB) + MariaDB — 변경 불가 |
| C-02 | API 하위 호환성 유지 | 기존 클라이언트 변경 없이 서버 측만 개선 |
| C-03 | 현행 배포 구조 | 단일 코드베이스, 역할별 인스턴스, DB 공유 — 설계 시작점 |
| C-04 | 점진적 적용 | 전체 재작성 불가, 기존 코드베이스와 공존 방식 |
| C-05 | 프로토타입 외부 서버 제약 | 외부 연계 서버 Stub/Mock 대체, 핵심 기능만 구현 |
| C-06 | 검증 환경 규모 제약 | 2만 명 물리 환경 불가, 축소 시뮬레이션으로 대체 |

---

#### C-01. 현행 기술 스택 고정

> 언어·프레임워크·DB 엔진 변경은 본 설계의 범위에서 제외한다.

현행 미팅 포털 서버는 Java Spring Boot 기반으로 구현되어 있으며, 본 설계 및 프로토타입은 이를 기반으로 한다.

- 언어: Java (Spring Boot 3.x)
- 커넥션 풀: HikariCP
- HTTP 클라이언트: Feign (동기, connect timeout 1,000ms / read timeout 3,000ms 기본값. Hystrix Feign 설정으로 외부 서버별 개별 적용 가능)
- 서킷 브레이커: Hystrix (Spring Cloud Netflix — 유지보수 중단, Spring Boot 3.x 공식 지원 종료)
- RDBMS: MariaDB

---

#### C-02. API 하위 호환성 유지

> 기존 클라이언트 변경 없이 서버 측 변경만으로 개선을 적용해야 한다.

기존 클라이언트(PC 클라이언트, 모바일 앱, 회의실 장비)와의 API 하위 호환성을 반드시 유지해야 한다. 본 설계에서 도입하는 아키텍처 개선은 클라이언트 배포를 요구하지 않는 방식으로 적용되어야 한다.

- 기존 REST API 경로, HTTP 메서드, 요청/응답 스키마 변경 불가
- 클라이언트 배포 없이 서버 측 변경만으로 개선 적용

---

#### C-03. 현행 배포 구조

> front-api / server-api / admin-api는 단일 코드베이스에서 역할별 별도 인스턴스로 배포되며 동일 DB를 공유하는 상태다. 설계는 이 구조를 시작점으로 한다.

본 설계의 개선 대상이 되는 현행 구조는 다음과 같다.

- 단일 코드베이스, 역할별 인스턴스 분리 배포
- front-api / server-api / admin-api가 동일 DB를 공유
- 환경별(Knox / Brity 대외 / Brity 공공 행망·공망) 분기 처리가 코드베이스 내에 누적

---

#### C-04. 점진적 적용

> 전체 서비스를 일시에 재작성하는 방식이 아닌, 기존 코드베이스와 공존 가능한 방식으로 점진적으로 적용해야 한다.

비동기화, 캐시 도입, 커넥션 풀 분리 등의 개선은 기존 동작을 유지하면서 단계적으로 적용 가능한 구조로 설계한다.

---

#### C-05. 프로토타입 외부 서버 제약

> 아키텍처 검증 프로토타입에서 외부 연계 서버는 Stub/Mock으로 대체한다.

WC서버, VC서버, AC서버, AI서버, Copilot Admin 서버 등 실제 외부 연계 서버와의 연동은 프로토타입 범위에서 제외한다. 핵심 아키텍처 문제(스레드·커넥션 풀 고갈, 우선순위 제어, 장애 격리 등)를 증명하는 기능만 구현하며, 보안·로깅·모니터링 등 운영 기능은 검증에 필요한 최소 수준만 구현한다.

---

#### C-06. 검증 환경 규모 제약

> 실제 2만 명 규모의 물리적 부하 환경은 보장되지 않는다. 단일 서버 시뮬레이션으로 대체할 수 있다.

성능 검증은 k6 등의 부하 테스트 도구를 사용하여 시나리오 기반으로 수행하며, 2만 명 동시 입장 시나리오는 단일 서버 환경에서의 축소 시뮬레이션으로 대체한다.
