### 5.2. 아키텍처 구현

5.1의 검증 시나리오를 실제로 수행하기 위한 구현 산출물과 측정 구성을 정리한다. front-api가 설계 전략(AS-01~09)을 실현한 코드 구성은 4.2 실행·모듈 뷰에서 다뤘으므로, 본 절은 검증에 필요한 (가) 소프트웨어 스택, (나) 적용 전후(A/B) 구성, (다) 지표 측정·관측 구성에 집중한다.

프로토타입은 C-01 기술 스택(Java Spring Boot, HikariCP, Feign, MariaDB) 안에서 구현하며, EOL된 Hystrix만 Resilience4j로 현대화한다(AS-09).

#### 5.2.2 적용 전후(A/B) 검증 구성

검증은 같은 부하를 전략 적용 전(A, As-is 재현)과 적용 후(B, To-be)에 걸어 결과 속성을 정량 비교하는 A/B 방식이다. 적용 전은 해당 전략만 비활성화해 현행 구조를 재현하며, Spring profile과 설정 값으로 토글한다. IV-04는 외부 장애를 주입한 상태에서 CB 유무를 비교하는 사건 주입형 A/B다.

| IV | 적용 전(A) 재현 | 적용 후(B) | 비교 대상 전략 |
| :---: | ----- | ----- | :---: |
| IV-01 | @Async 해제 · 단일 Connector · 공유 풀 | @Async · 8081 전용 Connector · join-pool | AS-02·04·08 |
| IV-02 | 단일 공유 풀 | 기능별 풀 분리(join/service/query) | AS-08 |
| IV-03 | 캐시 비활성(매 요청 외부 호출) | L1/L2 캐시 + Pre-warming | AS-03·05 |
| IV-04 | CB 비활성(Feign 직접 호출) + 장애 주입 | Resilience4j CB + 동일 장애 주입 | AS-09 |
| IV-05 | 단일 Primary(readOnly도 Primary) | Primary/Replica 라우팅 | AS-07 |

#### 5.2.3 지표 측정·관측

부하는 k6로 발생시키고, 애플리케이션 지표는 Micrometer가 노출한 `/actuator/prometheus`를 Prometheus가 수집해 Grafana로 시각화한다. 판정에는 k6 요약(성공률·p50/p95·실패율)과 HikariCP 풀 사용률·Resilience4j CB 상태·캐시 hit율을 함께 사용한다. 5.3의 각 그래프는 이 관측 파이프라인에서 캡처한다.
