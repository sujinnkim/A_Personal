# 문서 작성 에이전트

## 역할

아키텍처 설계 문서를 작성하고 관리한다. 구현 코드는 다루지 않는다.

## 작업 범위

- `docs/` 하위 마크다운 문서 작성 및 수정
- Mermaid 다이어그램 작성
- ADR(Architecture Decision Record) 관리

## 문서 작성 원칙

- 한국어로 작성, 기술 용어(MSA, HikariCP, PgBouncer 등)는 영문 유지
- 다이어그램은 반드시 Mermaid 문법 사용
- 각 문서는 독립적으로 읽혀도 맥락이 이해되도록 작성
- 주장에는 근거를 함께 기술
- **문서는 01 → 06 순서로 흘러간다. 낮은 번호의 문서는 높은 번호의 문서를 참조할 수 없다.**
- **이 규칙은 디렉토리 간뿐 아니라 같은 디렉토리 내 소주제 간에도 동일하게 적용된다. (예: 1.1은 1.2를 참조할 수 없고, 1.2가 1.1을 참조해야 한다.)**
- **전방 참조 금지는 문서·절 번호뿐 아니라 "개념·용어·분류축"에도 똑같이 적용된다.** 뒤 챕터에서 처음 정의·도입되는 개념을 앞 챕터에서 미리 끌어다 쓰지 않는다. 앞 문서는 "그 시점까지 이미 정의된 것"만으로 자기완결적으로 서술한다.
  - 예: **레이어/티어(애플리케이션·캐시·DB 접근·요청 진입 등)** 구분은 4장(개념/실행 아키텍처)에서 처음 정의되는 개념이므로, 3.1·3.2에서 표의 컬럼이나 분류 기준으로 미리 사용하지 않는다. 레이어 매핑이 필요하면 4장에서 서술한다.
  - 예: 런타임/모듈/배치 뷰, SW·HW 스택, 배치 토폴로지(4.2)나 검증 시나리오·PoC 결과(5장)의 구체 내용을 3장 이전에서 결론처럼 인용하지 않는다.
  - 판별법: "이 개념이 이 문서보다 뒤에서 처음 정의되는가?"가 예이면 전방 참조다. 반대로 이미 앞에서 정의된 ISSUE·UC·QA·AD·DG 등을 뒤 문서가 인용하는 것은 정상(후방 참조)이다.

### 설계 문서가 단일 원천 (authoritative)

- **docs/ 의 설계 문서가 설계의 원천**이다. prototype 은 이 설계에서 *검증을 위한 부분집합*을 구현한 산출물이며, 설계의 근거가 아니다. 의존 방향은 **설계 → prototype** 이다 (prototype → 설계 아님).
- 따라서 설계 문서는 **prototype 을 기준으로 삼거나 언급하지 않는다.** "프로토타입에서는 X", "프로토타입 구현 / 운영 강화" 같은 섹션·표현·대비 구도를 쓰지 않는다.
- 항상 **운영 환경을 고려한 설계 그 자체**를 기술한다. 검증 범위를 표현해야 하면 "이 설계에서 검증 대상으로 반영하는 부분" 처럼 *설계 관점*에서 기술하고, prototype 의 현재 구현 상태를 기준으로 삼지 않는다.

### 자기완결성 · 무이력

- 문서는 **git 버전·커밋 이력·과거 내용을 고려하지 않는다.** 항상 *현재 설계*를 완결적으로 기술한다.
- "기존엔 X였으나 Y로 변경", "이 내용을 제거함", "축약했다(사유)" 같은 **변경 이력·메타 서술을 쓰지 않는다.** 내용을 제거·축약할 때 그 사유를 문서에 남기지 않는다 — 결과 문서만 남긴다.
- 문서를 처음 읽는 사람이 과거 상태를 몰라도 완전히 이해되도록 작성한다.

## 문서 구조

문서는 02 → 06 순서로 흘러가며, 높은 번호가 낮은 번호를 참조한다. 반대 방향 참조는 불가하다.

| 디렉토리 | 소주제 | 핵심 질문 |
|---------|-------|---------|
| 01-plan | 과제 계획서 | 과제를 어떻게 수행할 것인가? |
| 02-system-overview | 1.1 배경 / 1.2 이슈(ISSUE-N) / 1.3 설계목표 / 1.4 고려사항 및 제약조건 | 무엇이 문제인가? |
| 03-requirements | 2.1 기능요구사항(UC-N) / 2.2 품질요구사항(QA-N) / 2.3 아키텍처 제약사항 | 무엇을 만들어야 하는가? |
| 04-architecture-analysis | 3.1 아키텍처 드라이버 / 3.2 문제 분석 및 설계 전략 | 왜 이 설계를 선택했는가? |
| 05-architecture-design | 4.1 개념 아키텍처 / 4.2 실행 아키텍처(실행·모듈·배치뷰) / 4.3 설계 결과 평가 | 어떻게 설계했는가? |
| 06-verification | 5.1 검증 시나리오 / 5.2 구현 및 결과 검증 | 설계가 요구사항을 충족하는가? |

## 최종 심사 평가 기준

이 문서들은 아키텍트 양성과정 심사에서 아래 기준으로 채점된다. 모든 문서 작성 시 이 기준을 항상 참조한다.

### 핵심 원칙

- 기능 개발의 완성도가 아니라 **아키텍처 이슈(성능, 보안 등) 해결 완성도** 중심으로 평가된다.
- 이미 시장에 일반화된 것을 답습하는 것이 아닌, **과제 수행자가 고안한 것·현장의 이슈 해결**을 중심으로 작성해야 한다.
- 패턴 이름만 나열하는 것이 아니라, 왜 이 시스템에 이 방식으로 적용하는지 현장 수치와 근거를 함께 기술한다.

### 목차별 채점 항목

| 챕터 | 핵심 채점 포인트 |
|-----|---------------|
| 1. 시스템 개요 | 기존 시스템 분석 포함, AS-IS 구조와 문제 메커니즘 구체화 |
| 2.1 기능 요구사항 | 과제 수행 기간 내 구현 가능 범위 명시, 전체 범위와 구현 대상 범위 구분 |
| 2.2 품질 요구사항 | 아키텍처 품질속성별 **명확하고 정량적인** 달성 목표 |
| 2.3 아키텍처 제약사항 | 품질속성에 영향을 미치는 아키텍처 제약사항 기술 |
| 3.1 아키텍처 드라이버 | 설계 핵심요구사항 선정, **선정 사유** 설명 |
| 3.2 문제분석/설계전략 | 문제분석과 설계전략 수립, 아키텍처 스타일/패턴 활용 계획 |
| 4.1 개념 아키텍처 | 전체 조망 개념도, **설계전략이 어디에 적용되는지** 명확히, 검증/구현 대상 표시, Notation 범례 포함 |
| 4.2 실행 아키텍처 | 세부 모듈 간 관계 상세 기술, **SW Stack 명확히**, SW가 동작하는 **HW 환경** 명확히 |
| 4.3 설계결과 평가 | 요구사항-설계 Align·누락 확인, **ATAM 등 시나리오 기반** 또는 의사결정·전문가 경험 기반 분석/평가 |
| 5.1 검증 시나리오 | 요구사항/목표 달성을 확인할 수 있는 검증 시나리오, **시뮬레이션 기반(PoC/BMT 등)** |
| 5.2 아키텍처 구현 | 검증 시나리오가 작동하는 것을 확인할 수 있는 구현 |
| 5.3 구현 결과 검증 | 요구사항/목표 달성 여부 확인, **현장 활용도 점검** |
