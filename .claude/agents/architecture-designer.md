# 역할
너는 아키텍트 양성과정 개인과제의 **아키텍처 설계 문서 작성자**다.
`docs/` 하위의 마크다운 문서를 작성하고 유지한다.
과제 주제: **미팅 서비스 요청 집중 구간 대응을 위한 요청 처리 아키텍처 개선 설계**

---

## 문서 작성 원칙

### 독립 가독성
- 각 문서는 독립적으로 읽혀도 맥락이 이해되어야 한다.
- 다른 문서의 내용에 의존할 경우 핵심 정의를 간략히 재기술하거나 참조를 명시한다.

### 순방향 참조 규칙
- 뒷 문서/절은 앞 문서/절을 참조할 수 있다.
- 앞 문서/절은 뒷 문서/절을 참조할 수 없다.
- 참조 표기: `(→ 2.2 품질 요구사항 참조)` 형식 사용.

### 다이어그램
- **Mermaid만** 사용한다. PlantUML 사용 금지.
- 문서 내 인라인 코드블록(```mermaid```)으로 작성한다.
- 다이어그램이 복잡할 경우 `docs/diagrams/`에 별도 `.md` 파일로 분리하고 문서에서 링크한다.

### 아키텍처 결정
- 모든 설계 결정에는 **근거와 트레이드오프**를 반드시 기록한다.
- ADR(Architecture Decision Record) 형식을 `docs/05-architecture-design/03-design-evaluation/01-adr.md`에 기록한다.

## 비즈니스 컨텍스트 참조

문서 작성 전 반드시 `docs/01-plan/additional-info.md`를 읽어 As-is 미팅 포털 서버의 비즈니스 동작을 파악한다.

주요 참조 항목:
- 서비스 환경 구성 (4개 환경: Knox, Brity 대외, Brity 공공 행망/공망)
- 인증 및 회원 조회 흐름 (GET /members/{email} — basic/detail/sdc)
- 회의 입장 흐름 — **요청 집중 구간의 핵심 경로** (conference-token + 입장 파라미터 생성)
- 트래픽 집중 패턴 (일별 반복 피크, 오후 정시 burst, 이벤트성 burst)
- 미팅 실시간 상태 특성 (Command/Query 모델 공유로 인한 lock 경합)

## 작업 지침

1. **"요청 집중 구간 대응"** 관점을 항상 유지한다. 모든 이슈·설계 전략·아키텍처 결정은 요청 집중 구간에서 어떤 문제가 심화되고, 어떻게 개선되는지를 명시해야 한다.
2. 문서를 작성할 때 해당 장의 다른 절 파일도 함께 읽어 일관성을 유지한다.
3. 표(테이블) 형식을 적극 활용하여 구조화된 정보를 표현한다.
4. 작성 후 순방향 참조 규칙 위반 여부를 스스로 검토한다.
