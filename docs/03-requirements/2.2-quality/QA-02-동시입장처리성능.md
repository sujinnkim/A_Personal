|------|------|

## 시나리오 상세

### 현행 문제

- 입장 API는 WC서버에 OTP 발급을 동기 호출하며, 응답을 받는 동안 해당 요청의 스레드가 점유된다.
- 2만 건 동시 요청 시 2만 개 스레드가 WC서버 응답을 대기하게 되어 스레드 풀 고갈 위험이 있다.
- 스레드 고갈과 동시에 DB 커넥션 고갈이 발생하면 입장 처리 전체가 불가능해진다.
- 연계시스템A의 경우 1분간 2만 건 참석 처리가 필요하여 burst 규모가 더 크다.

### 개선 목표

- Virtual Thread(Java 21) 또는 WebFlux 논블로킹 처리로 스레드 점유 없이 다수의 동시 요청을 수용한다.
- 회의 공통 입장 정보(OTP 제외, 회의 메타데이터 등)를 Redis에 단기 캐시하여 DB 조회 횟수를 줄인다.
- WC서버 호출에 서버별 타임아웃을 격리 적용하여 특정 WC서버 지연이 전체 처리에 영향을 미치지 않도록 한다.
- 입장 처리 전용 DB 커넥션 풀을 다른 기능과 분리하여 커넥션 사용률을 관리한다.

### 측정 방법

| 측정 항목 | 기준값 | 측정 방법 |
|---------|------|---------|
| DB 커넥션 풀 사용률 | 80% 이하 | 부하 테스트 중 HikariCP 풀 모니터링 |
| 평균 응답시간 | 1초 이내 | 부하 테스트 (k6) — 2만 VU 동시 시나리오 |
| 스레드 사용량 | 기존 대비 현저히 감소 | JVM 스레드 수 모니터링 |
| 처리 성공률 | 99% 이상 | 부하 테스트 응답 코드 분석 |
