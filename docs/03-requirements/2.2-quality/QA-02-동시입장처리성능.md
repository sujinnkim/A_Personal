|------|------|

## 시나리오 상세

### 현행 문제

- 입장 API는 WC서버에 Feign 동기 호출하며, 응답을 받는 동안 해당 요청의 스레드가 점유된다.
- 2만 건 동시 요청 시 2만 개 스레드가 WC서버 응답을 대기하게 되어 스레드 풀 고갈 위험이 있다.
- 캐시 없이 매 요청마다 DB 조회가 발생하여 DB 커넥션 소비가 집중된다.
- 스레드 고갈과 커넥션 고갈이 동시에 발생하면 입장 처리 전체가 불가능해진다.
- 단일 커넥션 풀을 모든 기능이 공유하므로, 입장 처리 부하가 다른 기능의 DB 접근까지 차단한다.

### 개선 목표

- 요청 집중 구간에서 2만 명 동시 입장 요청을 처리할 때 DB 커넥션 풀 사용률 80% 이하를 유지한다.
- 요청 집중 구간에서도 평균 응답시간 1초 이내를 달성한다.
- 입장 처리 부하가 증가하더라도 다른 기능의 정상 처리에 영향을 미치지 않도록 한다.

### 측정 방법

| 측정 항목 | 기준값 | 측정 방법 |
|---------|------|---------|
| DB 커넥션 풀 사용률 | 80% 이하 | 부하 테스트 중 HikariCP 풀 모니터링 |
| 평균 응답시간 | 1초 이내 | 부하 테스트 (k6) — 2만 VU 동시 시나리오 |
| 처리 성공률 | 99% 이상 | 부하 테스트 응답 코드 분석 |
