# 2.2.3. QA-03. DB 커넥션 풀 격리 신뢰성

## 품질 속성 시나리오

| 항목 | 내용 |
|------|------|
| 품질속성 | 신뢰성 (장애 격리, Reliability - Fault Isolation) |
| 우선순위 | 상 |
| 자극 출처 | 입장 처리 기능의 burst 트래픽 |
| 자극 | 2만 명 동시 입장 처리로 인해 입장 전용 DB 커넥션 풀이 고갈 상태에 도달함 |
| 환경 | 정상 운영 중, 입장 처리 기능의 DB 커넥션 풀 고갈 발생 |
| 응답 | 입장 기능의 커넥션 풀 고갈에도 불구하고 회의 조회, 회의 개설 등 타 기능은 전용 커넥션 풀을 통해 정상적으로 DB에 접근하여 서비스 제공 |
| 측정 기준 | 입장 전용 커넥션 풀 고갈 상황에서 회의 조회 API(`GET /meetings`)의 응답 성공률 **100%** 유지 |

---

## 시나리오 상세

### 현행 문제

- 현재 DB 커넥션 풀이 기능 간 격리 없이 공유된다.
- 입장 처리 burst 발생 시 입장 관련 쿼리가 전체 커넥션을 소진한다.
- 커넥션 풀 고갈 이후에는 회의 조회, 회의 개설, 권한 갱신 등 모든 기능이 DB에 접근하지 못해 전체 서비스 장애로 전파된다.
- 커넥션 획득 대기 타임아웃 초과가 연쇄적으로 발생하여 오류 응답이 급증한다.

### 개선 목표

- HikariCP 멀티 DataSource 구성으로 기능 단위 커넥션 풀을 분리한다.
  - 입장 처리 전용 풀: 입장 파라미터 생성 관련 쿼리
  - 조회 전용 풀: 회의 목록/상세 조회 쿼리
  - 기타(개설/권한) 전용 풀: 회의 개설, 권한 갱신 등 나머지 쿼리
- 각 풀은 독립적인 최대 커넥션 수와 커넥션 획득 타임아웃을 갖는다.
- 입장 전용 풀이 고갈되어도 조회 전용 풀은 영향을 받지 않는다.

### 풀 분리 구성 예시

| 풀 이름 | 담당 기능 | 최대 커넥션 수 예시 | 획득 타임아웃 예시 |
|--------|---------|----------------|--------------|
| join-pool | 입장 처리 (OTP 관련 DB 작업) | 50 | 3초 |
| query-pool | 회의 조회 | 30 | 5초 |
| general-pool | 회의 개설, 권한 갱신, 기타 | 30 | 5초 |

### 측정 방법

| 측정 항목 | 기준값 | 측정 방법 |
|---------|------|---------|
| 타 기능 응답 성공률 | 100% | 입장 풀 고갈 유도 후 조회 API 호출 성공 여부 확인 |
| 조회 응답시간 | 정상 응답시간 유지 | 장애 격리 시나리오 부하 테스트 |
| 커넥션 풀 고갈 전파 여부 | 전파 없음 | HikariCP 풀별 메트릭 모니터링 |
