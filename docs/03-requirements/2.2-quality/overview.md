# 2.2. 품질 요구사항

본 절은 1.2 이슈, 1.3 설계 목표, 2.1 기능 요구사항이 요구하는 시스템 품질을 정량화한 결과이다. 각 품질 요구사항은 **ISO/IEC 25010** 품질 모델로 분류하고, 어떤 사용자 시나리오에서 검증하는지, 어떤 정량 기준으로 합격을 판정하는지를 함께 밝힌다.

본 절에서 다루는 품질 요구사항은 **아키텍처 스타일과 무관하게 시스템이 만족해야 할 품질**에 한정한다. 특정 아키텍처 결정에서 비롯되는 추가 품질 요구는 3장 아키텍처 분석에서 그 결정과 함께 도출한다.

## 중요도·난이도 기준

| 등급 | 중요도 기준 | 난이도 기준 |
|------|-----------|-----------|
| 상 | 미달성 시 핵심 비즈니스 목표(DG) 자체가 훼손됨 | 아키텍처 수준의 구조적 결정 필요, 복수 설계 전략 조합 및 검증 복잡 |
| 중 | 미달성 시 운영 품질이 저하되나 핵심 기능은 유지됨 | 명확한 패턴(Bulkhead, Circuit Breaker 등) 적용으로 해결 가능 |
| 하 | 미달성 시 사용 편의에 영향을 주나 서비스는 유지됨 | 표준 설정 변경 수준으로 해결 가능 |

각 QA의 중요도·난이도는 위 등급 기준을 유래 이슈(1.2절)에 대입하여 판정한다. 판정 근거는 다음과 같다.

| QA | 유래 이슈 | 중요도 근거 (미달 시 훼손 대상) | 난이도 근거 |
|:---:|:---:|------|------|
| QA-01 | ISSUE-02 | 로그인은 전 사용자의 서비스 진입점으로, 미달 시 전체 사용자 경험이 훼손됨 (상) | 외부 서버 `allOf` 대기 구조를 캐시 기반으로 전환하는 구조적 결정 필요 (상) |
| QA-02 | ISSUE-01 | 8만 명 동시 입장 실패는 핵심 서비스 목적을 직접 훼손 (상) | 동기 블로킹 처리를 비동기·Bulkhead 구조로 재설계해야 함 (상) |
| QA-03 | ISSUE-04 | 특정 기능의 커넥션 고갈이 전체 서비스로 전파되면 가용성 훼손 (상) | Bulkhead 패턴 적용으로 해결 가능 (중) |
| QA-04 | 복합(ISSUE-01·04·05·06·07·09) | 핵심 기능 성공률은 서비스 품질의 최종 지표 (상) | 비동기·캐시·Bulkhead·CB·선제대응 등 복수 전략을 조합·검증해야 함 (상) |
| QA-05 | ISSUE-05·06 | 외부 서버 한 곳의 장애가 무관 기능까지 전파되면 전체 가용성 훼손 (상) | Circuit Breaker 패턴 적용으로 해결 가능 (중) |

## 핵심·보조 드라이버 분류

```
핵심 드라이버 (중요도·난이도 모두 상)
   QA-01 시간 응답성 (권한 갱신)
   QA-02 처리량 (동시 입장)
   QA-04 가용성 (핵심 기능 성공률)

보조 드라이버 (중요도 상, 난이도 중)
   QA-03 결함 허용성 (커넥션 풀 격리)
   QA-05 결함 허용성 (외부 서버 장애 격리)
```

QA-01·02·04는 본 과제의 핵심 주제인 "요청 집중 구간 대응"을 직접 정량화한 결과로, 미달성 시 서비스 목적이 곧바로 훼손된다. QA-03·05는 미달성 시 영향이 동일하나 명확한 패턴(Bulkhead, Circuit Breaker)으로 해결 가능하여 난이도가 상대적으로 낮다. 모든 QA는 3.1 아키텍처 드라이버에서 직접 인용된다.

품질 요구사항은 성격에 따라 두 층으로 구분된다. **QA-04(핵심 기능 성공률)는 다른 품질 요구사항이 함께 충족될 때 달성되는 종합 결과 지표**이고, **QA-01(응답시간)·QA-02(처리량)·QA-03(내부 커넥션 풀 격리)·QA-05(외부 서버 장애 격리)는 각각 독립된 원인·메커니즘 기준**이다. 특히 QA-03과 QA-05는 각각 "내부 자원 격리"와 "외부 연계 격리"라는 서로 다른 결함 격리 축을 담당하며, QA-04의 하위 항목이 아니라 QA-04를 떠받치는 독립 요구사항이다.
