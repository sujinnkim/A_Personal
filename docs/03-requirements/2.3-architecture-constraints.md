# 2.3. 아키텍처 제약사항

아키텍처 제약사항은 설계 선택지를 제한하는 사전 결정 조건이다. 1장의 고려사항(→ 1.4 고려사항 및 제약조건 참조)을 기반으로 구체적인 제약 항목을 정의하며, 각 제약은 설계 결정에 직접 영향을 미친다.

## 2.3.1. 제약사항 목록

| ID | 유형 | 제약사항 | 근거 | 영향 |
|----|------|---------|------|------|
| CON-01 | 기술 제약 | 구현 언어는 Java, 프레임워크는 Spring Boot 3.x를 사용한다. | 현행 시스템 기술 스택 유지 | 비동기 처리 구현 방식은 Spring 생태계 내 옵션(Virtual Thread, WebFlux, CompletableFuture)으로 제한됨 |
| CON-02 | 기술 제약 | 데이터베이스는 MariaDB를 사용하며, 커넥션 풀은 HikariCP를 사용한다. | 현행 시스템 DB 스택 유지 | 커넥션 풀 격리는 HikariCP 멀티 DataSource로 구현 |
| CON-03 | 기술 제약 | 현행 모놀리식 구조를 기반으로 설계하며, 서비스 분리 시 적용할 목표 인프라로 Kubernetes + Istio를 고려한다. | 현행 시스템 운영 구조 및 향후 전환 방향 | Pod 수평 확장은 HPA로 처리하는 방향을 목표로 한다 |
| CON-04 | 운영 제약 | 기존 클라이언트(PC 클라이언트, 모바일 앱, 회의실 장비)의 API 하위 호환성을 유지한다. | 클라이언트 배포 없이 서버 측 변경 적용 필요 | API 경로, HTTP 메서드, 요청/응답 스키마 변경 불가. 내부 처리 방식만 변경 |
| CON-05 | 구현 범위 | 프로토타입은 아키텍처 핵심 검증에 필요한 기능만 구현한다. 외부 연계 서버는 Stub/Mock으로 대체한다. | 과제 범위 및 구현 일정 제약 | WC/VC/AC/AI/Copilot 서버와의 실제 연동은 구현하지 않으며, 응답 지연 등 행동을 Mock으로 시뮬레이션 |

## 2.3.2. 제약사항 간 관계

- CON-01, CON-02는 비동기 처리, 커넥션 풀 격리 등 핵심 설계 전략의 구현 기술 선택지를 제한한다. 비동기 처리는 Spring 생태계 내 옵션(Virtual Thread, WebFlux, CompletableFuture)으로, 커넥션 풀 격리는 HikariCP 멀티 DataSource 방식으로 구현하게 된다.
- CON-03은 현행 모놀리식 구조를 전제로 설계하되, 서비스 분리 시 Kubernetes + Istio 기반 인프라 적용을 목표 방향으로 명시한다. 따라서 설계 결정은 현행 구조에서 적용 가능한 개선 방안을 우선으로 하며, 서비스 분리 이후의 인프라 활용 방안도 함께 고려한다.
- CON-04는 입장 집중 구간 개선 방식을 결정할 때 클라이언트 변경이 필요한 요청 큐잉 방식을 배제하고 서버 내부 논블로킹 처리를 선택하는 근거가 된다.
- CON-05는 프로토타입 구현 범위를 결정하며, 아키텍처 핵심 검증에 집중하도록 범위를 한정한다.
