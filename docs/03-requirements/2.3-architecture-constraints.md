# 2.3. 아키텍처 제약사항

본 절은 품질 요구사항(2.2)을 충족하는 아키텍처 솔루션의 **탐색 공간을 좁히는 기술 제약**을 기술한다. 1.4의 제약조건(C-01~C-06)이 "확정된 사실"을 서술한다면, 본 절은 그 사실들의 **기술적 파급**과 기술 요소 자체의 한계를 아키텍처 결정에 직접 작용하는 형태로 정리한다. 각 제약은 특정 해결 방안을 배제하거나 설계 방향을 강제한다.

| ID | 분류 | 설명 | 근거 | 관련 QA |
|----|------|------|------|---------|
| CR-01 | 기술 제약 | Spring MVC + JDBC 동기 블로킹 모델에서 외부 서버 호출·커넥션 대기 중 해당 요청의 스레드가 점유 상태로 유지된다. 비동기·리액티브 전환(R2DBC, WebFlux 등) 없이는 이 구조적 특성을 우회할 수 없으며, 스레드 풀 고갈 방지를 위한 설계가 이 제약 안에서 이루어져야 한다. | C-01 (Java Spring Boot + HikariCP 고정) | QA-02, QA-04 |
| CR-02 | 기술 제약 | 기능별 커넥션 풀 분리(Bulkhead) 적용 시 동일 DB에 연결되는 총 커넥션 수가 증가한다. DB 서버의 최대 커넥션 수 상한을 초과하지 않도록 각 풀의 크기 합산을 설계 시 반드시 고려해야 하며, 풀 크기 합산 통제가 설계 결정의 제약 조건이 된다. | C-01 (HikariCP 고정), C-03 (DB 공유 구조) | QA-02, QA-03 |
| CR-03 | 기술 제약 | 현행 단일 코드베이스·DB 공유 구조를 기반으로 설계하며, 전면 재작성 없이 기존 코드베이스와 공존 가능한 방식으로 점진적으로 적용해야 한다. 서비스 분리가 필요한 경우에도 기존 동작을 유지하면서 단계적으로 전환할 수 있는 설계를 우선한다. | C-03 (현행 배포 구조), C-04 (점진적 적용) | QA-03, QA-04 |
| CR-04 | 운영 제약 | 기존 클라이언트(PC 클라이언트, 모바일 앱, 회의실 장비)의 API 하위 호환성을 유지해야 한다. 클라이언트 측 변경이 필요한 방식(요청 큐잉 프로토콜 변경, 새 엔드포인트 강제 전환 등)은 설계 선택지에서 제외된다. 내부 처리 방식의 변경만 허용된다. | C-02 (API 하위 호환성 유지) | QA-02, QA-04 |
| CR-05 | 구현 범위 | 프로토타입은 아키텍처 핵심 검증에 필요한 기능만 구현한다. 외부 연계 서버(WC/VC/AC/AI/Copilot 서버 등)는 Stub/Mock으로 대체하며, 실제 연동 환경에서의 추가 동작은 설계 문서로 명시한다. | C-05 (프로토타입 외부 서버 제약) | — |
| CR-06 | 기술 제약 | 현행 외부 서버 호출은 Feign(동기, read timeout 3,000ms)으로 구현되어 있으며, 응답 대기 중 해당 스레드가 점유 상태를 유지한다. 외부 서버 장애·응답 지연 시 스레드가 최대 3,000ms 묶이며, VC + AC 순차 호출 시 최대 6,000ms 고정이 가능하다. 이 특성이 설계에서 해결해야 할 구조적 제약이 된다. | C-01 (Feign 동기 호출 고정) | QA-02, QA-04, QA-05 |
| CR-07 | 기술 제약 | 다수의 포털 서버 인스턴스가 각각 로컬 캐시를 보유하는 구조에서는 인스턴스 간 캐시 불일치가 구조적으로 발생한다. 권한·설정 데이터의 실시간 정합성이 요구되는 경우, 인스턴스 간 캐시 불일치를 허용 가능한 수준으로 관리하기 위한 TTL 정책과 캐시 무효화 전략이 설계에 포함되어야 한다. | C-01 (Spring Boot 고정), C-03 (다중 인스턴스 배포 구조) | QA-01, QA-04 |

## 제약사항 간 관계

- **CR-01 · CR-06**은 스레드 점유 문제의 두 원천이다. CR-01은 JDBC 커넥션 대기 중 스레드 점유를, CR-06은 Feign 외부 호출 대기 중 스레드 점유를 각각 강제한다. 두 제약이 동시에 작용하는 피크 구간(동시 입장 + 외부 서버 호출 집중)에서 스레드 고갈 위험이 최고조에 달한다.
- **CR-02**는 CR-01(HikariCP 고정)과 CR-03(DB 공유 구조)의 결합 파급이다. 기능별 풀 분리(Bulkhead)를 적용하면 격리는 달성되지만 총 커넥션 수가 늘어 DB 서버 상한 초과 위험이 생긴다. 풀 크기 합산 설계가 항상 동반되어야 한다.
- **CR-03**은 CR-01·CR-02·CR-06의 개선 방식을 "기존 코드베이스 공존" 범위로 제한한다. 전면 재작성이나 운영 중단 전환을 요구하는 방식은 설계 선택지에서 배제된다.
- **CR-04**는 입장 집중 구간 개선 방식을 결정할 때 클라이언트 변경이 필요한 방식을 배제한다. 요청 완충·우선순위 제어는 서버 내부에서만 이루어져야 한다.
- **CR-07**은 캐시 설계에서 분산 캐시(Redis)와 로컬 캐시(Caffeine)의 적용 범위를 결정하는 제약이 된다. 정합성 요구가 높은 데이터는 분산 캐시, 낮은 데이터는 로컬 캐시로 분리하는 방향을 강제한다.
