# 2.1.6. UC-06. 회의 취소

## 유스케이스 개요

| 항목 | 내용 |
|------|------|
| Use Case ID | UC-06 |
| Use Case명 | 회의 취소 |
| 관련 기능 요구사항 | FR-06 |
| 관련 이슈 | ISSUE-05, ISSUE-06 |
| 행위자 | 미팅 사용자 (회의 개설자), 미팅 클라이언트 (PC 앱 / 모바일 앱) |
| 사전조건 | 로그인 완료 (유효한 액세스 토큰 보유), 취소 대상 회의가 존재하며 사용자가 개설자 또는 취소 권한 보유, 회의가 아직 시작되지 않은 상태 |
| 사후조건 | 회의 상태가 '취소'로 변경됨. 외부 서버(WC/VC/AC)에 회의 삭제가 Feign 동기 호출로 전달됨. |

---

## 기본 흐름

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 1 | 미팅 사용자 | 회의 상세 화면에서 회의 취소 요청 |
| 2 | 미팅 클라이언트 | `DELETE /meetings/{meetingId}` 또는 `PATCH /meetings/{meetingId}/cancel` 요청 전송 |
| 3 | front-api | 토큰 유효성 검증 |
| 4 | front-api | 요청 유효성 검증 (취소 권한, 취소 가능한 상태 여부 확인) |
| 5 | front-api | DB에서 회의 상태를 '취소'로 업데이트 (소프트 딜리트) |
| 6 | front-api 또는 server-api | 외부 서버(WC/VC/AC)에 회의 삭제 요청 Feign 동기 호출로 전달. WC 전용 회의는 WC서버에만, VC/AC 포함 회의는 server-api를 통해 해당 벤더 연동 서버에 전파 |
| 7 | WC서버 / VC서버 / AC서버 | 각 서버에서 회의 삭제 처리 후 결과 반환 |
| 8 | front-api | 외부 서버 삭제 완료 확인 후 취소 완료 응답 반환 |
| 9 | front-api | 참석자 취소 알림 발송 (동기 처리) |
| 10 | 미팅 클라이언트 | 취소 완료 화면 표시 |

## 대안 흐름

### A1. 외부 서버 삭제 처리 실패

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 6a | front-api 또는 server-api | WC서버 또는 VC서버 또는 AC서버 삭제 요청 실패 (5xx, read timeout 3,000ms 만료) |
| 7a | front-api | 장애 격리 구조 없음. 외부 서버 오류가 전체 취소 요청 실패로 전파됨. 5xx 반환 |
| 비고 | | DB에서 '취소' 상태로 이미 변경된 경우 외부 서버와 상태 불일치 발생 가능 |

### A2. 이미 진행 중인 회의 취소 시도

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 4a | front-api | 회의 상태 확인 결과 진행 중 상태 확인 |
| 5a | front-api | 409 Conflict 반환 (진행 중 회의는 취소 불가, 종료 처리 안내 포함) |

### A3. 취소 권한 없음

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 4b | front-api | 요청 사용자가 취소 권한 없음 확인 |
| 5b | front-api | 403 Forbidden 반환 |

### A4. 연계시스템A 회의 대규모 취소

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 2c | 연계시스템A | 다수 회의에 대한 취소 요청 순차 전송 |
| 5c | front-api | 건별 DB 상태 업데이트 (개별 트랜잭션) |
| 6c | front-api 또는 server-api | 각 취소 건에 대해 외부 서버에 Feign 동기 호출로 삭제 요청 순차 전달 |
| 비고 | | 대규모 취소 시 요청 수에 비례하여 스레드 점유가 누적됨 |

## 시스템 인터페이스

| API | HTTP 메서드 | 경로 | 설명 |
|-----|-----------|------|------|
| 회의 취소 | DELETE | /meetings/{meetingId} | 회의 취소 처리 및 외부 서버 삭제 전파 |
| 회의 상태 변경 | PATCH | /meetings/{meetingId}/cancel | 회의 상태를 '취소'로 명시적 변경 |

---

## 아키텍처 관련 주목 사항

이 유스케이스는 ISSUE-05, ISSUE-06의 발현 경로다.

- 단계 6에서 외부 서버(WC/VC/AC) 삭제를 Feign 동기 호출로 수행한다. 외부 서버 응답을 받는 동안 스레드가 점유 상태로 대기하며, 이 지연이 취소 요청의 응답 지연으로 직접 연결된다. VC + AC 순차 호출 시 최대 6,000ms 스레드가 고정된다(→ ISSUE-05).
- 외부 서버 장애 시 Circuit Breaker 등 장애 격리 구조 없이 전체 취소 요청이 실패한다. DB에서 '취소' 상태로 변경된 이후 외부 서버 전파 단계에서 실패할 경우 DB 상태와 외부 서버 상태 간 불일치가 발생할 수 있다(→ ISSUE-06).
- 단계 9의 참석자 취소 알림 발송도 동기 처리로 수행되어, 알림 발송 지연이 취소 응답 시간에 포함된다. 참석자 수가 많을수록 알림 발송 지연이 길어져 취소 응답 시간에 직접 영향을 미친다.
- 연계시스템A를 통한 대규모 일괄 취소 시 각 취소 건에 대해 외부 서버 동기 호출을 순차 수행하므로, 취소 건수에 비례하여 처리 시간이 선형으로 증가한다.
