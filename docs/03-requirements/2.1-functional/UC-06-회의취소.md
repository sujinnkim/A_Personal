# 2.1.6. UC-06. 회의 취소

## 유스케이스 개요

| 항목 | 내용 |
|------|------|
| Use Case ID | UC-06 |
| Use Case명 | 회의 취소 |
| 관련 기능 요구사항 | FR-06 |
| 행위자 | 미팅 사용자 (회의 개설자), 미팅 클라이언트 (PC 앱 / 모바일 앱) |
| 사전조건 | 로그인 완료 (유효한 액세스 토큰 보유), 취소 대상 회의가 존재하며 사용자가 개설자 또는 취소 권한 보유, 회의가 아직 시작되지 않은 상태 |
| 사후조건 | 회의 상태가 '취소'로 변경됨. 외부 서버(WC/VC/AC)에서 해당 회의가 삭제 처리됨. 참석자에게 취소 알림이 비동기 발송됨. |

---

## 기본 흐름

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 1 | 미팅 사용자 | 회의 상세 화면에서 회의 취소 요청 |
| 2 | 미팅 클라이언트 | `DELETE /meetings/{meetingId}` 또는 `PATCH /meetings/{meetingId}/cancel` 요청 전송 |
| 3 | Gateway | 토큰 유효성 검증 후 Meeting Service로 라우팅 |
| 4 | Meeting Service | 요청 유효성 검증 (취소 권한, 취소 가능한 상태 여부 확인) |
| 5 | Meeting Service | DB에서 회의 상태를 '취소'로 업데이트 (소프트 딜리트, 개설 전용 DB 커넥션 풀 사용) |
| 6 | Meeting Service | 취소 완료 응답 즉시 반환 |
| 7 | Meeting Service | 외부 서버(WC/VC/AC) 회의 삭제 이벤트를 비동기 큐에 적재 |
| 8 | Event Consumer | WC서버, VC서버, AC서버에 회의 삭제 요청 비동기 전달 |
| 9 | WC서버, VC서버, AC서버 | 각 서버에서 회의 삭제 처리 |
| 10 | Meeting Service | 참석자 취소 알림 이벤트를 별도 비동기 큐에 적재 |
| 11 | Notification Consumer | 참석자 전원에게 회의 취소 알림 발송 |

---

## 대안 흐름

### A1. 외부 서버 삭제 처리 실패

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 8a | Event Consumer | WC서버 또는 VC서버 또는 AC서버 삭제 요청 실패 (5xx, 타임아웃) |
| 9a | Event Consumer | Circuit Breaker 동작. 실패한 삭제 이벤트를 Dead Letter Queue에 적재 |
| 10a | Event Consumer | 재시도 스케줄러가 Dead Letter Queue를 처리하여 결과적 일관성 확보 |
| 연속 | | DB의 회의 상태는 이미 '취소'로 변경되어 있어 서비스 정합성 유지 |

### A2. 이미 진행 중인 회의 취소 시도

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 4a | Meeting Service | 회의 상태 확인 결과 진행 중 상태 확인 |
| 5a | Meeting Service | 409 Conflict 반환 (진행 중 회의는 취소 불가, 종료 처리 안내 포함) |

### A3. 취소 권한 없음

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 4b | Meeting Service | 요청 사용자가 취소 권한 없음 확인 |
| 5b | Meeting Service | 403 Forbidden 반환 |

### A4. 연계시스템A 회의 대규모 취소

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 2c | 연계시스템A | 다수 회의에 대한 일괄 취소 요청 전송 |
| 5c | Meeting Service | 배치 처리로 DB 일괄 상태 업데이트 (개별 트랜잭션 분리) |
| 7c | Meeting Service | 외부 서버 삭제 이벤트를 배치 단위로 큐에 적재하여 외부 서버 부하 평준화 |

## 시스템 인터페이스

| API | HTTP 메서드 | 경로 | 설명 |
|-----|-----------|------|------|
| 회의 취소 | DELETE | /meetings/{meetingId} | 회의 취소 처리 및 외부 서버 삭제 이벤트 발행 |
| 회의 상태 변경 | PATCH | /meetings/{meetingId}/cancel | 회의 상태를 '취소'로 명시적 변경 |

## 아키텍처 관련 주목 사항

- 단계 6에서 DB 상태 변경 완료 즉시 응답을 반환하고, 외부 서버 삭제(단계 7~9)는 비동기 이벤트 큐를 통해 처리한다. 이는 외부 서버의 삭제 지연이 취소 응답 시간에 영향을 미치지 않도록 하기 위한 비동기 분리(DS-05) 전략이다.
- 외부 서버 삭제 실패는 Dead Letter Queue와 재시도 스케줄러를 통해 결과적 일관성(Eventual Consistency)을 보장한다. DB의 취소 상태가 단일 정합성 기준이 된다.
- 참석자 알림 발송(단계 10~11)은 취소 처리 완료 경로와 완전히 분리된 별도 비동기 큐로 처리하여, 대규모 알림 발송이 취소 처리 성능에 영향을 미치지 않도록 한다.
- 연계시스템A를 통한 대규모 일괄 취소 시, 외부 서버 삭제 이벤트를 Rate Limiting을 적용하여 배치 단위로 큐에 적재함으로써 피크 시간대 외부 서버 과부하를 방지한다.
