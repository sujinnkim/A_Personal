# 2.1.3. UC-03. 회의 입장

## 유스케이스 개요

| 항목 | 내용 |
|------|------|
| Use Case ID | UC-03 |
| Use Case명 | 회의 입장 |
| 관련 기능 요구사항 | FR-02 |
| 행위자 | 미팅 사용자, 미팅 클라이언트 (PC 앱 / 모바일 앱 / 회의실 장비) |
| 사전조건 | 로그인 완료 (유효한 액세스 토큰 보유), 입장하려는 회의가 존재함 |
| 사후조건 | 입장 파라미터(OTP 포함) 반환 완료. 사용자가 WC서버에 접속하여 회의에 참여함. |

---

## 기본 흐름 (웹 컨퍼런스 입장)

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 1 | 미팅 사용자 | 클라이언트에서 회의 입장 버튼 클릭 |
| 2 | 미팅 클라이언트 | `GET /meetings/{meetingId}/join` 요청 전송 |
| 3 | Gateway | 토큰 유효성 검증 후 Meeting Service로 라우팅 |
| 4 | Meeting Service | 사용자 권한 검증 (입장 허용 여부) |
| 5 | Meeting Service | 회의 공통 입장 정보 캐시 조회 (Redis, 단기 TTL) |
| 5a | Meeting Service | [캐시 HIT] 캐시에서 공통 입장 정보 즉시 조회 |
| 5b | Meeting Service | [캐시 MISS] DB에서 회의 정보 조회 후 캐시 저장 |
| 6 | Meeting Service | WC서버에 OTP 발급 요청 (논블로킹 또는 타임아웃 격리 동기 호출) |
| 7 | WC서버 | OTP 생성 및 반환 |
| 8 | Meeting Service | 입장 파라미터 조합 (공통 입장 정보 + OTP) |
| 9 | Meeting Service | 입장 파라미터 응답 반환 |
| 10 | 미팅 클라이언트 | 입장 파라미터를 사용하여 WC서버에 직접 접속 |
| 11 | 미팅 사용자 | 회의 화면 진입 |

## 대안 흐름

### A1. 2만 명 동시 입장 (Burst 시나리오)

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 1a | 2만 명의 미팅 사용자 | 동시에 입장 버튼 클릭 |
| 2a | 미팅 클라이언트 | 2만 건의 `GET /meetings/{meetingId}/join` 요청 동시 전송 |
| 4a | Meeting Service | 논블로킹 처리(Virtual Thread)로 스레드 점유 없이 요청 접수 |
| 5a | Meeting Service | 대부분의 요청이 캐시 HIT → DB 조회 최소화 |
| 6a | Meeting Service | WC서버별 타임아웃 격리 적용, 과부하 시 다른 지역 WC서버로 failover |
| 9a | Meeting Service | 순차적으로 입장 파라미터 응답 (일부 지연 가능하나 실패 없음) |

### A2. WC서버 응답 지연

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 7b | WC서버 | 설정된 타임아웃(예: 2초) 내 OTP 발급 응답 없음 |
| 8b | Meeting Service | 타임아웃 초과 시 다른 지역 WC서버로 재시도 (Retry with backoff) |
| 9b | Meeting Service | 모든 WC서버 타임아웃 초과 시 503 반환 및 재시도 안내 |

### A3. 사용자 입장 권한 없음

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 4c | Meeting Service | 권한 캐시 조회 결과 해당 사용자는 해당 회의 입장 불가 |
| 5c | Meeting Service | 403 Forbidden 반환 |

### A4. 전화 참석 (오디오 컨퍼런스 입장)

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 2d | 미팅 클라이언트 | 전화 참석 파라미터 조회 요청 |
| 6d | Meeting Service | AC서버에 오디오 컨퍼런스 전화 참석 처리 요청 |
| 7d | AC서버 | 전화 참석 정보(전화번호, PIN 등) 반환 |
| 9d | Meeting Service | 전화 참석 파라미터 응답 반환 |

## 성능 요구사항 연계

이 유스케이스는 본 설계의 핵심 성능 검증 대상이다.

| 조건 | 목표 |
|------|------|
| 2만 명 동시 요청 | DB 커넥션 풀 사용률 80% 이하 |
| 2만 명 동시 요청 | 평균 응답시간 1초 이내 |
| 처리 중 장애 | 입장 서비스 커넥션 고갈 시 조회 서비스 정상 동작 |

## 아키텍처 관련 주목 사항

- 단계 6의 WC서버 OTP 동기 호출이 ISSUE-02의 핵심 문제 구간이다. 스레드 점유로 인한 스레드 고갈 위험이 있다.
- Virtual Thread(Java 21)를 활용한 논블로킹 처리(DS-02)로 동시 처리 수용량을 대폭 향상한다.
- 단계 5의 입장 공통 정보 단기 캐시(DS-02)로 동일 회의에 대한 반복적인 DB 조회를 최소화한다.
- OTP는 사용자별 고유값으로 캐시 불가하며, WC서버 호출은 불가피하다.
- 입장 처리용 DB 커넥션 풀은 기타 기능과 격리된 전용 풀을 사용한다(DS-03).
- 이 유스케이스는 논블로킹 처리 및 DB 커넥션 풀 격리(벌크헤드) 전략의 핵심 검증 대상이다.
