## 대안 흐름

### A1. WC서버 응답 지연

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 7a | front-api | WC서버가 read timeout(3,000ms) 이내에 응답하지 않음. 스레드가 최대 3,000ms 점유 상태로 대기 |
| 8a | front-api | 타임아웃 만료 후 예외 처리. 장애 격리 구조 없음 |
| 9a | front-api | 503 반환. 사용자 입장 실패 |

### A2. 다수 사용자 동시 입장 (요청 집중)

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 2a | 다수의 미팅 사용자 | 동시에 입장 버튼 클릭, 대규모 `GET /meetings/{meetingId}/join` 요청 동시 유입 |
| 4a | front-api | 각 요청마다 DB에서 회의 정보를 개별 조회. 요청 수에 비례하여 DB 커넥션 소비 |
| 7a | front-api | 각 요청이 WC서버 Feign 동기 호출 동안 스레드 점유. 동시 요청 수에 비례하여 스레드 고갈 위험 증가 |
| 연속 | front-api | 스레드 풀 고갈 또는 DB 커넥션 풀 고갈 발생 시 신규 요청에 대한 응답 자체가 불가능해짐 |

### A3. 사용자 입장 권한 없음

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 4c | front-api | 권한 검증 결과 해당 사용자는 해당 회의 입장 불가 확인 |
| 5c | front-api | 403 Forbidden 반환 |

### A4. 전화 참석 (오디오 컨퍼런스 입장)

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 2d | 미팅 웹 | 전화 참석 파라미터 조회 요청 |
| 6d | front-api | AC서버에 오디오 컨퍼런스 전화 참석 처리 요청 (Feign 동기 호출, read timeout 3,000ms) |
| 7d | AC서버 | 전화 참석 정보(전화번호, PIN 등) 반환 |
| 9d | front-api | 전화 참석 파라미터 응답 반환 |

## 아키텍처 관련 주목 사항

이 유스케이스는 요청 집중 구간의 핵심 경로이며, ISSUE-01, ISSUE-03, ISSUE-07의 핵심 발현 경로다.

- 단계 7의 WC서버 Feign 동기 호출은 ISSUE-01의 핵심 문제 구간이다. WC서버 응답을 기다리는 동안 해당 요청의 스레드가 최대 3,000ms 점유 상태로 유지된다. 2만 명 동시 입장 요청이 집중되면 스레드 풀 고갈과 DB 커넥션 풀 고갈이 동시에 발생할 수 있다(→ ISSUE-01).
- 단계 5에서 캐시 없이 매 요청마다 DB에서 회의 정보를 개별 조회한다. 동일 회의에 수십~수천 명이 동시 입장하는 상황에서도 각 요청이 DB 커넥션을 개별 소비한다. 단일 HikariCP 커넥션 풀을 다른 모든 기능과 공유하므로, 입장 처리의 DB 부하가 회의 조회·개설 등 다른 기능에 영향을 미친다(→ ISSUE-03, ISSUE-04).
- 요청 집중 시간대(오후 정시 burst, 이벤트성 burst)에 사용자 입장 처리(front-api → WC서버)와 cPaaS 피드백(퇴장·연결 끊김 등, cPaaS → server-api 경유)이 동일 DB 테이블에 동시에 write를 발생시킨다. Command/Query가 동일 DB·도메인 모델을 공유하므로 write/read 충돌과 lock 경합이 심화된다(→ ISSUE-07).
- 요청 집중 구간에서 회의 입장 처리(conference-token 발급 + 입장 파라미터 전달)가 최우선 처리 대상이나, 현재 아키텍처에는 요청 우선순위를 구분하는 구조가 없다. 단순 조회 등 덜 중요한 요청과 동일 스레드 풀·커넥션 풀을 경쟁하여 사용한다(→ ISSUE-03).
