# 2.1.5. UC-05. 회의 수정

## 유스케이스 개요

| 항목 | 내용 |
|------|------|
| Use Case ID | UC-05 |
| Use Case명 | 회의 수정 |
| 관련 기능 요구사항 | FR-05 |
| 관련 이슈 | ISSUE-05, ISSUE-06 |
| 행위자 | 미팅 사용자 (회의 개설자), 미팅 클라이언트 (PC 앱 / 모바일 앱) |
| 사전조건 | 로그인 완료 (유효한 액세스 토큰 보유), 수정 대상 회의가 존재하며 사용자가 개설자 또는 수정 권한 보유 |
| 사후조건 | 수정된 회의 정보가 DB에 반영됨. 관련 외부 서버(WC/VC/AC)에 변경 사항이 Feign 동기 호출로 전파됨. |

---

## 기본 흐름

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 1 | 미팅 사용자 | 회의 상세 화면에서 수정 항목(제목, 일정, 참석자, 회의 옵션 등) 변경 후 저장 요청 |
| 2 | 미팅 클라이언트 | `PATCH /meetings/{meetingId}` 요청 전송 (변경 항목만 포함) |
| 3 | front-api | 토큰 유효성 검증 |
| 4 | front-api | 요청 유효성 검증 (수정 권한, 파라미터, 이미 종료된 회의 여부 등) |
| 5 | front-api | DB에서 현재 회의 정보 조회 (변경 전 상태 확인) |
| 6 | front-api | DB에 변경된 회의 정보 저장 |
| 7 | front-api 또는 server-api | 변경된 항목에 따라 외부 서버(WC/VC/AC)에 변경 사항 Feign 동기 호출로 전파. WC 전용 회의는 WC서버에만, VC/AC 포함 회의는 server-api를 통해 해당 벤더 연동 서버에 전파 |
| 8 | WC서버 / VC서버 / AC서버 | 회의 정보 업데이트 후 결과 반환 |
| 9 | front-api | 외부 서버 전파 완료 후 수정 완료 응답 반환 |
| 10 | front-api | 참석자 변경 알림 발송 (동기 처리) |
| 11 | 미팅 클라이언트 | 수정 완료 화면 표시 |

## 대안 흐름

### A1. VC서버 / AC서버 변경 포함 수정

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 7a | server-api | 회의 유형에 따라 VC서버, AC서버에 변경 사항 Feign 동기 순차 호출로 전파 (최대 6,000ms 스레드 고정 가능) |
| 8a | VC서버, AC서버 | 각각 업데이트 처리 후 결과 반환 |
| 9a | front-api | 모든 외부 서버 전파 완료 후 수정 완료 응답 반환 |

### A2. 외부 서버 전파 실패

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 7b | front-api 또는 server-api | 외부 서버(WC/VC/AC)가 read timeout(3,000ms) 내 응답하지 않거나 오류 반환 |
| 8b | front-api | 장애 격리 구조 없음. 외부 서버 오류가 전체 수정 요청 실패로 전파됨. 5xx 반환 |
| 비고 | | DB에 저장된 변경 정보와 외부 서버 상태 간 불일치 발생 가능 |

### A3. 수정 권한 없음

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 4a | front-api | 요청 사용자가 개설자가 아니거나 수정 권한 없음 확인 |
| 5a | front-api | 403 Forbidden 반환 |

### A4. 이미 진행 중 또는 종료된 회의 수정 시도

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 4b | front-api | 회의 상태 확인 결과 수정 불가 상태(진행 중 / 종료) |
| 5b | front-api | 409 Conflict 반환 (수정 불가 상태 안내 메시지 포함) |

## 시스템 인터페이스

| API | HTTP 메서드 | 경로 | 설명 |
|-----|-----------|------|------|
| 회의 수정 | PATCH | /meetings/{meetingId} | 회의 정보 부분 수정 및 외부 서버 변경 전파 |
| 회의 전체 수정 | PUT | /meetings/{meetingId} | 회의 정보 전체 갱신 |

## 아키텍처 관련 주목 사항

이 유스케이스는 ISSUE-05, ISSUE-06의 핵심 발현 경로다.

- 단계 7에서 외부 서버(WC/VC/AC) 변경 전파를 Feign 동기 호출로 수행한다. 외부 서버 응답을 받는 동안 스레드가 점유 상태로 대기하며, 이 응답 지연이 수정 요청의 응답 지연으로 직접 연결된다. VC + AC 순차 호출 시 최대 6,000ms 스레드가 고정된다(→ ISSUE-05).
- 외부 서버(WC/VC/AC) 장애 시 Circuit Breaker 등 장애 격리 구조 없이 전체 수정 요청이 실패한다. DB에 수정 정보가 저장된 이후 외부 서버 전파 단계에서 실패할 경우 DB 상태와 외부 서버 상태 간 불일치가 발생할 수 있다(→ ISSUE-06).
- 요청 집중 시간대에 개설·수정 요청이 동일 서블릿 스레드 풀과 HikariCP 커넥션 풀을 공유하여 사용한다. 외부 서버 호출 중 스레드 점유가 중첩되면 스레드 풀 고갈이 빠르게 발생한다.
- 단계 10의 참석자 변경 알림 발송도 동기 처리로 수행되어, 알림 발송 지연이 수정 응답 시간에 포함된다.
