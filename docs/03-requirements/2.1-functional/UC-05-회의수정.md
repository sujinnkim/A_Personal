# 2.1.5. UC-05. 회의 수정

## 유스케이스 개요

| 항목 | 내용 |
|------|------|
| Use Case ID | UC-05 |
| Use Case명 | 회의 수정 |
| 관련 기능 요구사항 | FR-05 |
| 행위자 | 미팅 사용자 (회의 개설자), 미팅 클라이언트 (PC 앱 / 모바일 앱) |
| 사전조건 | 로그인 완료 (유효한 액세스 토큰 보유), 수정 대상 회의가 존재하며 사용자가 개설자 또는 수정 권한 보유 |
| 사후조건 | 수정된 회의 정보가 DB에 반영됨. 관련 외부 서버(WC/VC/AC)에 변경 사항이 전파됨. 참석자에게 변경 알림이 발송됨. |

---

## 기본 흐름

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 1 | 미팅 사용자 | 회의 상세 화면에서 수정 항목(제목, 일정, 참석자, 회의 옵션 등) 변경 후 저장 요청 |
| 2 | 미팅 클라이언트 | `PATCH /meetings/{meetingId}` 요청 전송 (변경 항목만 포함) |
| 3 | Gateway | 토큰 유효성 검증 후 Meeting Service로 라우팅 |
| 4 | Meeting Service | 요청 유효성 검증 (수정 권한, 파라미터, 이미 종료된 회의 여부 등) |
| 5 | Meeting Service | DB에서 현재 회의 정보 조회 (변경 전 상태 확인) |
| 6 | Meeting Service | DB에 변경된 회의 정보 저장 (개설 전용 DB 커넥션 풀 사용) |
| 7 | Meeting Service | 변경된 항목에 따라 외부 서버(WC/VC/AC)에 변경 사항 전파 (비동기 병렬 처리) |
| 8 | WC서버 | 웹 컨퍼런스 정보 업데이트 후 결과 반환 |
| 9 | Meeting Service | 수정 완료 응답 반환 (외부 서버 전파 결과와 무관하게 즉시 응답) |
| 10 | Meeting Service | 참석자 변경 알림 비동기 발송 (이벤트 큐 적재) |
| 11 | 미팅 클라이언트 | 수정 완료 화면 표시 |

---

## 대안 흐름

### A1. VC서버 / AC서버 변경 포함 수정

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 7a | Meeting Service | 회의 유형에 따라 WC서버 외 VC서버, AC서버에도 변경 사항 비동기 병렬 전파 |
| 8a | VC서버, AC서버 | 각각 업데이트 처리 후 결과 반환 |
| 9a | Meeting Service | 모든 외부 서버 전파 완료 여부와 무관하게 DB 저장 완료 기준으로 즉시 응답 |
| 연속 | | 외부 서버 전파 실패는 재시도 큐(Dead Letter Queue)를 통해 별도 처리 |

### A2. 외부 서버 전파 실패

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 7b | Meeting Service | 외부 서버(WC/VC/AC)가 타임아웃 또는 오류 반환 |
| 8b | Meeting Service | Circuit Breaker 동작. 실패한 전파 이벤트를 재시도 큐에 적재 |
| 9b | Meeting Service | DB 저장 성공 기준으로 수정 완료 응답 반환 (부분 성공 처리) |
| 10b | Meeting Service | 외부 서버 전파 실패 알림 로그 기록 및 재시도 스케줄러 동작 |

### A3. 수정 권한 없음

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 4a | Meeting Service | 요청 사용자가 개설자가 아니거나 수정 권한 없음 확인 |
| 5a | Meeting Service | 403 Forbidden 반환 |

### A4. 이미 진행 중 또는 종료된 회의 수정 시도

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 4b | Meeting Service | 회의 상태 확인 결과 수정 불가 상태(진행 중 / 종료) |
| 5b | Meeting Service | 409 Conflict 반환 (수정 불가 상태 안내 메시지 포함) |

## 시스템 인터페이스

| API | HTTP 메서드 | 경로 | 설명 |
|-----|-----------|------|------|
| 회의 수정 | PATCH | /meetings/{meetingId} | 회의 정보 부분 수정 및 외부 서버 변경 전파 |
| 회의 전체 수정 | PUT | /meetings/{meetingId} | 회의 정보 전체 갱신 |

---

## 아키텍처 관련 주목 사항

- 단계 7의 외부 서버(WC/VC/AC) 변경 전파는 응답 필수 경로에서 분리되어 비동기 처리된다. DB 저장 완료 시점에 즉시 응답함으로써 외부 서버 지연이 수정 응답 시간에 영향을 미치지 않도록 한다.
- 외부 서버 전파 실패는 Circuit Breaker(DS-05)와 Dead Letter Queue 조합으로 처리한다. 실패한 이벤트는 재시도 큐를 통해 결과적 일관성(Eventual Consistency)을 보장한다.
- 단계 10의 참석자 변경 알림은 이벤트 큐를 통한 비동기 발송으로 처리하여 알림 지연이 수정 응답에 영향을 미치지 않도록 한다.
- 회의 수정 요청은 피크 시간대에 개설 요청과 동일한 외부 서버를 공유하므로, 외부 서버 호출에 커넥션 풀 격리(DS-03)를 적용하여 수정 요청이 개설 요청의 커넥션 고갈로 차단되지 않도록 한다.
