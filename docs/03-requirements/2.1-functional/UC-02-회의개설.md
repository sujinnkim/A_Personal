# 2.1.2. UC-02. 회의 개설

## 유스케이스 개요

| 항목 | 내용 |
|------|------|
| Use Case ID | UC-02 |
| Use Case명 | 회의 개설 |
| 관련 기능 요구사항 | FR-03 |
| 행위자 | 미팅 사용자, 미팅 클라이언트 (PC 앱 / 모바일 앱 / 연계시스템) |
| 사전조건 | 로그인 완료 (유효한 액세스 토큰 보유) |
| 사후조건 | 회의가 개설됨. 회의 정보가 DB에 저장됨. 해당 회의 유형에 맞는 외부 서버(WC/VC/AC)에 회의가 생성됨. |

---

## 기본 흐름 (웹 컨퍼런스 개설)

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 1 | 미팅 사용자 | 회의 개설 정보(제목, 일정, 참석자, 회의 유형 등) 입력 및 개설 요청 |
| 2 | 미팅 클라이언트 | `POST /meetings` 요청 전송 (액세스 토큰 포함) |
| 3 | Gateway | 토큰 유효성 검증 후 Meeting Service로 라우팅 |
| 4 | Meeting Service | 회의 개설 요청 유효성 검증 (권한, 파라미터 등) |
| 5 | Meeting Service | 로컬 캐시(Caffeine)에서 회의 설정 정책 조회 |
| 5a | Meeting Service | [캐시 HIT] 캐시에서 정책 정보 즉시 조회 |
| 5b | Meeting Service | [캐시 MISS] DB에서 설정 정책 조회 후 로컬 캐시 저장 |
| 6 | Meeting Service | WC서버에 웹 컨퍼런스 개설 요청 (동기 호출, 타임아웃 적용) |
| 7 | WC서버 | 웹 컨퍼런스 생성 및 회의 ID 반환 |
| 8 | Meeting Service | 회의 정보 DB 저장 (입장 서비스 전용 커넥션 풀 사용) |
| 9 | Meeting Service | 응답에 필수적이지 않은 후처리(통계, 알림 등) 비동기 처리 |
| 10 | Meeting Service | 개설된 회의 정보(회의 ID, 입장 링크 등) 응답 반환 |
| 11 | 미팅 클라이언트 | 개설 완료 화면 표시 |

## 대안 흐름

### A1. VC서버 포함 회의 개설 (화상회의실 장비 포함)

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 6a | Meeting Service | WC서버 호출과 함께 VC서버에도 비디오 컨퍼런스 개설 요청 (비동기 병렬 또는 순차) |
| 7a | VC서버 | 비디오 컨퍼런스 생성 및 결과 반환 |
| 연속 | | 이후 단계는 기본 흐름과 동일 |

### A2. 전화 참석 포함 회의 개설

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 6b | Meeting Service | AC서버에 오디오 컨퍼런스 개설 요청 (비동기 분리 가능) |
| 7b | AC서버 | 오디오 컨퍼런스 생성 및 전화 참석 정보 반환 |
| 연속 | | 이후 단계는 기본 흐름과 동일 |

### A3. 외부 서버 응답 지연 / 오류

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 7c | Meeting Service | 외부 서버(WC/VC/AC)가 타임아웃 내 응답하지 않거나 오류 반환 |
| 8c | Meeting Service | Circuit Breaker 동작 또는 타임아웃 fallback 처리 |
| 9c | Meeting Service | 503 Service Unavailable 또는 부분 성공 응답 반환 |

### A4. 연계시스템C를 통한 통합 회의 개설

|------|--------|------|

| 3d | Open API Service | 요청 변환 후 Meeting Service로 전달 |

## 아키텍처 관련 주목 사항

- 단계 6의 WC서버 동기 호출은 ISSUE-04의 핵심 문제 구간이다. 피크 시간대에 WC서버 응답 지연이 전체 회의 개설 응답 시간에 직접 영향을 미친다.
- 단계 5의 로컬 캐시 적용(DS-04)으로 자주 변경되지 않는 설정 정책 조회에 대한 외부 DB 호출을 최소화한다.
- 단계 9의 비동기 분리를 통해 응답 필수 경로에서 불필요한 동기 처리를 제거한다.
- 이 유스케이스는 로컬 캐시 도입 및 응답 불필수 구간 비동기화 전략의 핵심 검증 대상이다.
