# 2.1.2. UC-02. 회의 개설

## 유스케이스 개요

| 항목 | 내용 |
|------|------|
| Use Case ID | UC-02 |
| Use Case명 | 회의 개설 |
| 관련 기능 요구사항 | FR-03 |
| 관련 이슈 | ISSUE-05, ISSUE-06, ISSUE-08 |
| 행위자 | 미팅 사용자, 미팅 클라이언트 (PC 앱 / 모바일 앱 / 연계시스템) |
| 사전조건 | 로그인 완료 (유효한 액세스 토큰 보유) |
| 사후조건 | 회의가 개설됨. 회의 정보가 DB에 저장됨. WC 전용 회의는 Meeting Manager 레벨에서 완결됨. VC/AC 포함 회의는 server-api를 통해 벤더 연동 서버에 회의가 생성됨. |

## 기본 흐름 (WC 전용 회의 개설)

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 1 | 미팅 사용자 | 회의 개설 정보(제목, 일정, 참석자, 회의 유형 등) 입력 및 개설 요청 |
| 2 | 미팅 클라이언트 | `POST /front/meetings` 요청 전송 (액세스 토큰 포함) |
| 3 | front-api | 토큰 유효성 검증 |
| 4 | front-api | 회의 개설 요청 유효성 검증, 미팅 타입 결정(WC), 권한 확인 (SDC 권한 DB 조회) |
| 5 | Meeting Manager | DB에서 회의 설정 정책 조회 |
| 6 | Meeting Manager | WC 전용 회의 정보 DB 저장 (server-api 호출 없이 완결) |
| 7 | Meeting Manager | 참석자 알림, 통계 처리 등 후처리를 동기로 수행 |
| 8 | front-api | 개설된 회의 정보(회의 ID, 입장 링크 등) 응답 반환 |
| 9 | 미팅 클라이언트 | 개설 완료 화면 표시 |

## 대안 흐름

### A1. VC/AC 포함 회의 개설 (server-api 경유)

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 4a | front-api | 미팅 타입 결정(VC 또는 AC 포함), 권한 확인 |
| 5a | Meeting Manager | 회의 기본 정보 DB 저장 후 server-api 호출 |
| 6a | server-api | VC 벤더 연동 서버에 비디오 컨퍼런스 개설 요청 (Feign 동기 호출, read timeout 3,000ms) |
| 7a | VC 벤더 연동 서버 | 비디오 컨퍼런스 생성 및 결과 반환 |
| 8a | server-api | AC 포함 시 AC 벤더 연동 서버에 오디오 컨퍼런스 개설 요청 (Feign 동기 순차 호출, read timeout 3,000ms). VC + AC 순차 호출 시 최대 6,000ms 스레드 고정 가능 |
| 9a | AC 벤더 연동 서버 | 오디오 컨퍼런스 생성 및 결과 반환 |
| 10a | Meeting Manager | 외부 연동 결과 DB 업데이트 후 응답 반환 |
| 비고 | | 외부 연동 실패(타임아웃·오류) 시 전체 회의 개설 요청 실패 처리 (5xx 반환) |

### A2. 외부 벤더 연동 서버 응답 지연 / 오류

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 6a 또는 8a | server-api | VC 또는 AC 벤더 연동 서버가 read timeout(3,000ms) 이내에 응답하지 않거나 오류 반환 |
| 연속 | server-api | 장애 격리 구조 없음. 타임아웃 만료 또는 오류 수신 즉시 전체 회의 개설 요청 실패 처리 (503 반환) |
| 연속 | Meeting Manager | DB에 저장된 회의 기본 정보를 롤백하거나 오류 상태로 처리 |

### A3. 연계시스템C를 통한 통합 회의 개설

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 2c | 연계시스템C | Open API Service를 통해 회의 개설 요청 전달 |
| 3c | Open API Service | 요청 변환 후 front-api로 전달 |
| 연속 | | 이후 단계는 기본 흐름과 동일 |

## 아키텍처 관련 주목 사항

이 유스케이스는 ISSUE-05, ISSUE-06, ISSUE-08의 핵심 발현 경로다.

- VC/AC 포함 회의 개설 시(대안 흐름 A1), server-api에서 벤더 연동 서버로 Feign 동기 호출을 순차 수행한다. VC + AC 순차 호출 시 최대 6,000ms(3,000ms × 2) 동안 스레드가 점유 상태로 고정된다. 요청 집중 시간대에 이러한 장기 스레드 점유가 중첩되면 스레드 풀 고갈로 이어진다(→ ISSUE-05).
- 외부 벤더 연동 서버 장애 시 Circuit Breaker 등 장애 격리 구조 없이 타임아웃 만료까지 스레드를 점유하다가 전체 회의 개설 요청이 실패한다. 외부 서버 1개의 장애가 회의 개설 기능 전체에 영향을 미친다(→ ISSUE-06).
- 연계시스템 증가에 따라 단일 코드베이스 내에 환경별·시스템별 분기 처리가 누적되고 있다. 새로운 연계 시스템 추가 시마다 기존 로직과의 결합이 심화된다(→ ISSUE-08).
- 단계 5에서 회의 설정 정책을 매 요청마다 DB에서 조회한다. 변경 빈도가 낮은 설정 정책임에도 서버사이드 캐시 없이 반복 조회가 발생한다.
