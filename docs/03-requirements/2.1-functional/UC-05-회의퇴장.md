# UC-05. 회의 퇴장

## 개요

인미팅 클라이언트에서 사용자가 퇴장하면 cPaaS를 통해 포털 server-api에 역방향으로
퇴장 이벤트가 전달되고, 참석자 상태가 DB에 반영된다.

## 행위자

- **주 행위자:** 인미팅 클라이언트 (via cPaaS)

## 사전 조건

- 회의 입장(UC-04)이 완료되어 인미팅 클라이언트가 실행 중인 상태이다.

## 사후 조건

- 참석자 상태가 DB에 '퇴장'으로 업데이트된다.

## 기본 흐름

1. 사용자가 인미팅 클라이언트에서 퇴장한다.
2. cPaaS가 포털 server-api에 퇴장 이벤트를 전달한다.
3. server-api가 참석자 상태를 DB에 업데이트한다.

## 예외 흐름

### 비정상 연결 끊김

1. 네트워크 단절 또는 클라이언트 강제 종료로 정상 퇴장 이벤트가 전달되지 않는다.
2. cPaaS가 연결 타임아웃 감지 후 연결 끊김 이벤트를 포털 server-api에 전달한다.
3. server-api가 참석자 상태를 '연결 끊김'으로 업데이트한다.

### DB write 경합

1. 대규모 미팅 시작 시점에 사용자 입장(write)과 cPaaS 피드백(write)이 동시에
   집중된다.
2. 단일 DB 모델을 공유하는 구조에서 참석자 상태 테이블에 lock 경합이 발생한다.
3. 입장 처리 및 퇴장 상태 반영이 함께 지연된다.
