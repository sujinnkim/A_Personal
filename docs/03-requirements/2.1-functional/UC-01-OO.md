# 2.1.1. UC-01. 로그인 및 권한 갱신

## 유스케이스 개요

| 항목 | 내용 |
|------|------|
| Use Case ID | UC-01 |
| Use Case명 | 로그인 및 권한 갱신 |
| 관련 기능 요구사항 | FR-01 |
| 행위자 | 미팅 사용자, 미팅 클라이언트 (PC 앱 / 모바일 앱) |
| 사전조건 | 미팅 웹/앱 접속 완료, 사용자 계정 보유 |
| 사후조건 | 로그인 완료 (액세스 토큰 발급), 권한 정보(AC 권한, LLM 권한, 용어사전 권한) 갱신 및 반환됨 |

---

## 기본 흐름

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 1 | 미팅 사용자 | 클라이언트에서 이메일/비밀번호를 입력하고 로그인 요청 |
| 2 | 미팅 클라이언트 | `POST /sign-in` 요청 전송 |
| 3 | Authentication Service | 자격증명 검증 후 JWT 액세스 토큰 발급 및 반환 |
| 4 | 미팅 클라이언트 | 액세스 토큰 수신 후 자동으로 `GET /members/{email}` 호출 |
| 5 | Meeting Service | 권한 캐시(Redis) 조회 |
| 6a | Meeting Service | [캐시 HIT] 캐시에서 권한 정보 즉시 반환 → 단계 9로 이동 |
| 6b | Meeting Service | [캐시 MISS] 외부 서버에 권한 갱신 요청 (비동기 병렬) |
| 7 | AC서버 | AC 권한 갱신 및 조회 결과 반환 |
| 7 | Copilot Admin 서버 | LLM 권한 갱신 및 조회 결과 반환 |
| 7 | Copilot Admin 서버 | 용어사전 권한 갱신 및 조회 결과 반환 |
| 8 | Meeting Service | 갱신된 권한 정보를 캐시(Redis)에 저장 |
| 9 | Meeting Service | 회원 정보 및 권한 정보를 포함한 응답 반환 |
| 10 | 미팅 클라이언트 | 응답 수신 후 사용자에게 서비스 화면 표시 |

---

## 대안 흐름

### A1. 외부 서버 응답 지연

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 7a | Meeting Service | 외부 서버(AC서버 또는 Copilot Admin 서버)가 설정된 타임아웃(예: 3초) 내에 응답하지 않음 |
| 8a | Meeting Service | 타임아웃된 서버의 권한은 기존 캐시 값을 유지 (stale 허용) 또는 기본값 적용 |
| 9a | Meeting Service | 부분적으로 갱신된 권한 정보와 함께 응답 반환 (성공 응답) |

### A2. 외부 서버 오류 (5xx, 연결 실패)

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 7b | Meeting Service | 외부 서버가 오류를 반환하거나 연결 실패 |
| 8b | Meeting Service | Circuit Breaker가 해당 서버에 대한 호출 차단 (Istio Circuit Breaker) |
| 9b | Meeting Service | 오류가 발생한 서버의 권한은 기존 캐시 값 유지, 나머지 권한으로 응답 반환 |
| 10b | Meeting Service | 오류 로그 기록, 권한 갱신 실패 이벤트 발행 |

### A3. 로그인 자격증명 오류

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 3a | Authentication Service | 자격증명 불일치 → 401 Unauthorized 반환 |
| 4a | 미팅 클라이언트 | 오류 메시지 표시, `GET /members/{email}` 호출 없음 |

## 시스템 인터페이스

| API | HTTP 메서드 | 경로 | 설명 |
|-----|-----------|------|------|
| 로그인 | POST | /sign-in | 자격증명 검증 및 JWT 발급 |
| 회원 정보 조회 | GET | /members/{email} | 권한 갱신 포함 회원 정보 반환 |

---

## 아키텍처 관련 주목 사항

- 단계 5-6b는 본 설계의 핵심 개선 대상이다. 개선 전에는 모든 외부 서버 응답을 `CompletableFuture.allOf()`로 동기 대기하여 전체 응답 시간이 가장 느린 서버에 의해 결정된다.
- 개선 후에는 캐시 HIT 시 외부 서버 호출 없이 즉시 응답하며, 캐시 MISS 또는 권한 변경 이벤트 수신 시에만 외부 서버를 호출하는 구조로 전환된다.
- 이 유스케이스는 권한 캐시 도입 및 이벤트 기반 갱신 전략의 핵심 검증 대상이다.
