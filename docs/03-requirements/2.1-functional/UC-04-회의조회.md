## 대안 흐름

### A1. 연계시스템A 대규모 조회

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 2a | 연계시스템A | 대량 회의 정보 조회 요청 |
| 4a | front-api | 인스턴스 내 단일 HikariCP 커넥션 풀에서 커넥션 획득. 연계시스템A 요청과 일반 사용자 요청이 동일 풀을 경쟁 사용 |
| 5a | front-api | DB에서 회의 정보 조회 (서버사이드 캐시 없이 매 요청마다 DB 직접 조회) |
| 6a | front-api | 조회 결과 반환 |

### A2. 입장 트래픽 집중 중 회의 조회 요청

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 2b | 미팅 사용자 | 다수 사용자 동시 입장 처리 중에 회의 목록 조회 요청 |
| 4b | front-api | 단일 HikariCP 커넥션 풀에서 커넥션 획득 시도. 입장 처리가 풀의 다수 커넥션을 점유 중이라면 커넥션 획득 대기 발생 |
| 연속 | front-api | 커넥션 획득 대기 시간 초과 시 조회 요청도 오류 응답 반환. 입장 처리 기능의 커넥션 고갈이 조회 기능 장애로 전파됨 |

## 아키텍처 관련 주목 사항

이 유스케이스는 ISSUE-03, ISSUE-04의 핵심 발현 경로다.

- front-api 인스턴스 내 모든 기능(회의 입장, 회의 조회, 회의 개설, 권한 갱신 등)이 단일 HikariCP 커넥션 풀을 공유한다. 요청 집중 시간대에 입장 처리가 커넥션 풀의 다수를 점유하면, 회의 조회 요청도 커넥션 획득 대기 → 타임아웃 → 오류로 이어진다. 기능 단위 커넥션 풀 격리가 없으므로 특정 기능의 트래픽 집중이 전체 서비스 장애로 전파된다(→ ISSUE-04).
- 회의 입장 처리(conference-token 발급 + 입장 파라미터 전달)는 요청 집중 구간에서 최우선 처리 대상이지만, 현재 아키텍처는 회의 조회와 회의 입장을 동일한 우선순위로 처리한다. 처리 우선순위 분리 구조가 없어 덜 중요한 조회 요청이 핵심 입장 처리 자원을 경쟁 소비한다(→ ISSUE-03).
- 서버사이드 캐시 없이 매 요청마다 DB를 직접 조회한다. 연계시스템A의 고빈도 조회와 일반 사용자 조회가 동일 DB·커넥션 풀을 경쟁 사용하여 피크 시간대 DB 부하가 가중된다.
- front-api, server-api, admin-api가 동일한 DB를 공유하므로 인스턴스 간 커넥션 수 조율이 없어 DB 서버 전체의 커넥션 수 누적 문제도 존재한다(→ ISSUE-04).
