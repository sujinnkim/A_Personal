## 대안 흐름

### A1. 연계시스템A 대규모 조회 (1분간 10만 건)

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 2a | 연계시스템A | 대량 회의 정보 조회 요청 (1분간 10만 건 규모) |
| 4a | System A Service | System A 전용 DB 커넥션 풀에서 커넥션 획득 (Meeting Service와 풀 격리) |
| 5a | System A Service | System A Cache 조회 후 캐시 MISS 시 DB 조회 |
| 6a | System A Service | 조회 결과 반환 |

### A2. 입장 서비스 커넥션 고갈 상황에서의 조회 (장애 격리 검증)

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 2b | 미팅 사용자 | 입장 서비스 커넥션 풀 고갈 중에 회의 목록 조회 요청 |
| 4b | Meeting Service | 조회 전용 커넥션 풀은 입장 전용 풀과 분리되어 있어 정상적으로 커넥션 획득 가능 |
| 6b | Meeting Service | 정상 응답 반환 (입장 서비스 장애와 무관하게 처리) |

### A3. 조회 결과 없음

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 5c | Meeting Service | 조건에 맞는 회의 정보 없음 |
| 6c | Meeting Service | 빈 목록 또는 404 응답 반환 |

## 성능 요구사항 연계

연계시스템A의 1분간 10만 건 조회는 별도의 System A Service 및 전용 캐시/DB를 통해 처리된다.

| 조건 | 목표 |
|------|------|
| 입장 서비스 커넥션 고갈 중 조회 요청 | 조회 서비스 정상 응답 (커넥션 풀 격리 효과) |
| 피크 시간대 동시 조회 | 조회 전용 커넥션 풀 내에서 안정적 처리 |

---

## 아키텍처 관련 주목 사항

- 이 유스케이스에서 가장 중요한 아키텍처 관심사는 DB 커넥션 풀 격리(DS-03)다. 입장 처리 burst로 인한 커넥션 고갈이 회의 조회에 영향을 미쳐서는 안 된다.
- 조회 전용 커넥션 풀(Meeting 조회용)과 입장 전용 커넥션 풀은 독립적으로 구성되어야 한다.
- 연계시스템A의 고빈도 조회(10만 건/분)는 System A Service + System A Cache를 통해 처리하며, Meeting Service의 부하와 격리된다.
- 이 유스케이스는 DB 커넥션 풀 격리(벌크헤드 패턴) 전략의 핵심 검증 대상이다.
