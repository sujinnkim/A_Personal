# 2.1.7. UC-07. 참석자 초대

## 유스케이스 개요

| 항목 | 내용 |
|------|------|
| Use Case ID | UC-07 |
| Use Case명 | 참석자 초대 |
| 관련 기능 요구사항 | FR-07 |
| 관련 이슈 | ISSUE-04, ISSUE-07 |
| 행위자 | 미팅 사용자 (회의 개설자 또는 초대 권한 보유자), 미팅 클라이언트 (PC 앱 / 모바일 앱) |
| 사전조건 | 로그인 완료 (유효한 액세스 토큰 보유), 초대 대상 회의가 존재하며 취소/종료 상태가 아님, 초대하려는 사용자 계정이 존재함 |
| 사후조건 | 초대된 참석자 정보가 DB에 저장됨. 초대 알림이 동기로 발송됨. 필요 시 외부 서버(WC/VC/AC)에 참석자 정보가 Feign 동기 호출로 전파됨. |

---

## 기본 흐름

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 1 | 미팅 사용자 | 회의 상세 화면에서 초대할 사용자를 검색 및 선택 후 초대 요청 |
| 2 | 미팅 클라이언트 | `POST /meetings/{meetingId}/attendees` 요청 전송 (초대 대상 사용자 목록 포함) |
| 3 | front-api | 토큰 유효성 검증 |
| 4 | front-api | 요청 유효성 검증 (초대 권한, 대상 사용자 존재 여부, 최대 참석자 수 초과 여부) |
| 5 | front-api | DB에 참석자 정보 저장. 인스턴스 내 단일 HikariCP 커넥션 풀 사용 (회의 입장·조회 등 다른 기능과 공유) |
| 6 | front-api | 외부 서버(WC/VC/AC)에 참석자 정보 업데이트 Feign 동기 호출로 전파 (회의 유형에 따라 선택적) |
| 7 | WC서버 / VC서버 / AC서버 | 참석자 정보 업데이트 처리 후 결과 반환 |
| 8 | front-api | 외부 서버 전파 완료 후 초대 알림 발송 (동기 처리) |
| 9 | front-api | 초대 완료 응답 반환 |
| 10 | 미팅 클라이언트 | 초대 완료 화면 표시 |

## 대안 흐름

### A1. 대규모 일괄 초대 (연계시스템A)

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 2a | 연계시스템A | 다수 회의에 대해 대량 참석자 초대 요청 전송 |
| 5a | front-api | 참석자 정보 순차 저장. 단일 HikariCP 커넥션 풀에서 커넥션 소비. 대량 요청 시 커넥션 고갈 위험 증가 |
| 6a | front-api | 각 초대 건에 대해 외부 서버 Feign 동기 호출 순차 수행 |
| 8a | front-api | 각 초대 건에 대해 알림 동기 발송 순차 수행 |

### A2. 이미 초대된 사용자 재초대 시도

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 4a | front-api | DB 조회 결과 해당 사용자가 이미 참석자 목록에 존재함 확인 |
| 5a | front-api | 409 Conflict 반환 (이미 초대된 사용자 목록 포함) |

### A3. 외부 서버 참석자 업데이트 실패

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 6a | front-api 또는 server-api | WC서버 또는 AC서버 참석자 업데이트 요청 실패 (5xx, read timeout 3,000ms 만료) |
| 7a | front-api | 장애 격리 구조 없음. 외부 서버 오류가 전체 초대 요청 실패로 전파됨. 5xx 반환 |
| 비고 | | DB에 저장된 참석자 정보와 외부 서버 상태 간 불일치 발생 가능 |

### A4. 초대 권한 없음

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 4b | front-api | 요청 사용자가 초대 권한 없음 확인 |
| 5b | front-api | 403 Forbidden 반환 |

## 아키텍처 관련 주목 사항

이 유스케이스는 ISSUE-04, ISSUE-07의 발현 경로다.

- 단계 5의 참석자 DB 저장이 인스턴스 내 단일 HikariCP 커넥션 풀을 사용한다. 요청 집중 시간대에 회의 입장 처리가 커넥션 풀의 다수를 점유하고 있으면, 참석자 초대 요청도 커넥션 획득 대기 → 타임아웃 → 오류로 이어질 수 있다. 특정 기능의 커넥션 고갈이 전체 서비스로 전파되는 구조다(→ ISSUE-04).
- 요청 집중 시간대에 사용자 입장 요청(front-api → WC서버)과 cPaaS 피드백(퇴장·연결 끊김, cPaaS → server-api 경유)이 동일 DB 테이블에 동시에 write를 발생시킨다. 참석자 초대(DB write)도 동일한 시점에 발생하므로, 대규모 미팅 시작 시점에 입장·피드백·초대 write가 동시에 발생해 DB lock 경합이 심화된다(→ ISSUE-07).
- 단계 6에서 외부 서버 참석자 업데이트를 Feign 동기 호출로 수행하며, 단계 8에서 알림 발송도 동기 처리한다. 외부 서버 응답 지연과 알림 발송 지연이 모두 초대 응답 시간에 누적된다. 외부 서버 장애 시 장애 격리 구조 없이 전체 초대 요청이 실패한다(→ ISSUE-05, ISSUE-06 연관).
- 대규모 일괄 초대 시 알림 발송이 동기 처리로 순차 수행되어 초대 건수에 비례하여 전체 처리 시간이 선형으로 증가한다.
