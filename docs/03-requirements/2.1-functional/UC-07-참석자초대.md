# 2.1.7. UC-07. 참석자 초대

## 유스케이스 개요

| 항목 | 내용 |
|------|------|
| Use Case ID | UC-07 |
| Use Case명 | 참석자 초대 |
| 관련 기능 요구사항 | FR-07 |
| 행위자 | 미팅 사용자 (회의 개설자 또는 초대 권한 보유자), 미팅 클라이언트 (PC 앱 / 모바일 앱) |
| 사전조건 | 로그인 완료 (유효한 액세스 토큰 보유), 초대 대상 회의가 존재하며 취소/종료 상태가 아님, 초대하려는 사용자 계정이 존재함 |
| 사후조건 | 초대된 참석자 정보가 DB에 저장됨. 초대 알림이 초대받은 사용자에게 비동기 발송됨. 필요 시 외부 서버(WC/VC/AC)에 참석자 정보 업데이트됨. |

---

## 기본 흐름

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 1 | 미팅 사용자 | 회의 상세 화면에서 초대할 사용자를 검색 및 선택 후 초대 요청 |
| 2 | 미팅 클라이언트 | `POST /meetings/{meetingId}/attendees` 요청 전송 (초대 대상 사용자 목록 포함) |
| 3 | Gateway | 토큰 유효성 검증 후 Meeting Service로 라우팅 |
| 4 | Meeting Service | 요청 유효성 검증 (초대 권한, 대상 사용자 존재 여부, 최대 참석자 수 초과 여부) |
| 5 | Meeting Service | DB에 참석자 정보 저장 (개설 전용 DB 커넥션 풀 사용) |
| 6 | Meeting Service | 초대 완료 응답 즉시 반환 |
| 7 | Meeting Service | 초대 알림 이벤트를 비동기 큐에 적재 |
| 8 | Notification Consumer | 초대받은 사용자에게 이메일 / 푸시 알림 발송 |
| 9 | Meeting Service | 외부 서버(WC/VC/AC) 참석자 업데이트 이벤트를 비동기 큐에 적재 (회의 유형에 따라 선택적) |
| 10 | Event Consumer | 해당 외부 서버에 참석자 정보 업데이트 요청 |

---

## 대안 흐름

### A1. 대규모 일괄 초대 (연계시스템A)

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 2a | 연계시스템A | 다수 회의에 대해 대량 참석자 초대 요청 전송 |
| 5a | Meeting Service | 배치 Insert로 참석자 정보 일괄 저장 (DB 커넥션 최소화) |
| 7a | Meeting Service | 알림 이벤트를 배치 단위로 큐에 적재하여 알림 서버 부하 평준화 |

### A2. 이미 초대된 사용자 재초대 시도

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 4a | Meeting Service | DB 조회 결과 해당 사용자가 이미 참석자 목록에 존재함 확인 |
| 5a | Meeting Service | 409 Conflict 반환 (이미 초대된 사용자 목록 포함) |

### A3. 외부 서버 참석자 업데이트 실패

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 10a | Event Consumer | WC서버 또는 AC서버 참석자 업데이트 요청 실패 |
| 11a | Event Consumer | 실패 이벤트를 Dead Letter Queue에 적재, 재시도 스케줄러 동작 |
| 연속 | | DB의 참석자 정보는 이미 저장 완료 상태. 결과적 일관성으로 처리 |

### A4. 초대 권한 없음

| 단계 | 행위자 | 설명 |
|------|--------|------|
| 4b | Meeting Service | 요청 사용자가 초대 권한 없음 확인 |
| 5b | Meeting Service | 403 Forbidden 반환 |

## 시스템 인터페이스

| API | HTTP 메서드 | 경로 | 설명 |
|-----|-----------|------|------|
| 참석자 초대 | POST | /meetings/{meetingId}/attendees | 회의에 참석자 추가 초대 및 알림 이벤트 발행 |
| 참석자 목록 조회 | GET | /meetings/{meetingId}/attendees | 현재 참석자 목록 반환 |

## 아키텍처 관련 주목 사항

- 단계 6에서 DB 저장 완료 즉시 응답을 반환하고, 알림 발송(단계 7~8)과 외부 서버 업데이트(단계 9~10)는 비동기 큐를 통해 처리한다. 이는 알림 서버 또는 외부 서버 지연이 초대 응답 시간에 영향을 미치지 않도록 하기 위한 비동기 분리(DS-05) 전략이다.
- 대규모 일괄 초대 시 알림 이벤트를 배치 단위로 큐에 적재하여 피크 시간대 알림 서버 과부하를 방지하는 Rate Limiting 전략을 적용한다.
- Virtual Thread(DS-02)를 활용하여 초대 요청이 집중되는 피크 시간대에도 스레드 점유 없이 다수의 초대 요청을 동시 처리할 수 있다.
- DB 저장(단계 5)과 외부 서버 업데이트(단계 9~10) 사이의 불일치는 결과적 일관성으로 허용하며, Dead Letter Queue와 재시도 스케줄러로 최종 일관성을 보장한다.
