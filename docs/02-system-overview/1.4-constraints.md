# 1.4. 고려사항 및 제약조건

본 설계를 수행함에 있어 반드시 준수해야 하는 제약조건과 사전에 고려해야 할 사항을 정리한다. 이들은 아키텍처 제약사항으로 구체화된다.

---

## 1.4.1. 기술 제약

### 언어 및 프레임워크

현행 미팅 포털 서버는 Java Spring Boot 기반으로 구현되어 있으며, 본 설계 및 프로토타입은 이를 기반으로 한다. 타 언어 또는 프레임워크로의 전환은 본 설계의 범위에서 제외한다.

- 언어: Java 21 (Virtual Thread 지원)
- 프레임워크: Spring Boot 3.x
- ORM: JPA / Spring Data JPA
- 커넥션 풀: HikariCP
- HTTP 클라이언트: Feign (동기, connect timeout 1,000ms / read timeout 3,000ms — 전체 외부 호출 일괄 적용)

### 데이터베이스

현행 데이터베이스는 MariaDB(RDBMS)이며, 캐시 레이어로 Redis를 활용한다. 데이터베이스 엔진 변경은 본 설계 범위에서 제외한다.

- RDBMS: MariaDB
- 캐시: Redis (분산 캐시), Caffeine (로컬 캐시)

### 인프라

현행 미팅 포털 서버는 단일 코드베이스로 구성된 모놀리식 구조이며, front-api / server-api / admin-api가 역할별 별도 인스턴스로 배포되고 동일 DB를 공유한다. 본 설계의 아키텍처 개선 전략은 MSA 전환을 포함한다.

- 현행 구조: 단일 코드베이스, 역할별 인스턴스 분리 배포, DB 공유
- 설계 목표 인프라: Kubernetes 기반 컨테이너 오케스트레이션, 서비스 메시(Istio) 도입 검토
- 자동 스케일링: HPA (Horizontal Pod Autoscaler)

---

## 1.4.2. 운영 제약

### API 하위 호환성 유지

기존 클라이언트(PC 앱, 모바일 앱, 회의실 장비)와의 API 하위 호환성을 반드시 유지해야 한다. 본 설계에서 도입하는 아키텍처 개선은 클라이언트 변경을 요구하지 않는 방식으로 적용되어야 한다.

- 기존 REST API 경로, HTTP 메서드, 요청/응답 스키마 변경 불가
- 클라이언트 배포 없이 서버 측 변경만으로 개선 적용

### 점진적 적용 가능성

전체 서비스를 일시에 재작성하는 방식이 아닌, 기존 서비스에 점진적으로 적용 가능한 설계를 우선한다. 특히 비동기화, 캐시 도입 등은 기존 코드베이스와 공존 가능한 방식으로 설계한다.

---

## 1.4.3. 구현 범위 제약

### 프로토타입 검증 범위

아키텍처 검증을 위한 프로토타입은 핵심 아키텍처 문제를 증명하는 기능만 구현한다. 실제 운영 서비스의 전체 기능을 구현하지 않는다.

- 구현 대상: 권한 캐시, 논블로킹 입장 처리, DB 커넥션 풀 격리, 로컬 캐시 + 타임아웃
- 외부 연계 서버(WC서버, VC서버, AC서버, Copilot Admin 서버 등)는 Stub/Mock으로 대체
- 보안(인증/인가), 로깅, 모니터링 등 운영 기능은 검증에 필요한 최소 수준만 구현

### 성능 검증 방법

성능 검증은 JMeter 또는 k6 등의 부하 테스트 도구를 사용하여 시나리오 기반으로 수행하며, 실제 2만 명 규모의 물리적 부하 환경은 단일 서버 시뮬레이션으로 대체할 수 있다.

## 1.4.4. 전제 및 가정

| 항목 | 가정 내용 | 비고 |
|------|---------|------|
| 인프라 전제 | Kubernetes + Istio는 MSA 전환 시 적용할 목표 인프라로 가정 | 실제 환경에서는 별도 구성 필요 |
| 외부 서버 | WC/VC/AC/AI/Copilot 서버는 Stub/Mock으로 대체 | 실제 연동 시 추가 설계 필요 |
| 트래픽 규모 | 2만 명 동시 입장 요청 집중을 최악의 시나리오로 가정 | 실제 분포는 스트리밍 서비스 오픈 후 관측 필요 |
| Redis 가용성 | Redis는 고가용성(Primary-Replica 또는 Cluster) 구성으로 가정 | 단일 Redis 장애 시의 fallback은 별도 설계 필요 |
| Java 버전 | Java 21 이상으로 가정 (Virtual Thread 활용 위해) | 기존 운영 Java 버전 확인 후 업그레이드 필요할 수 있음 |
| 현재 배포 구조 | front-api / server-api / admin-api가 단일 코드베이스에서 역할별 별도 인스턴스로 배포되며 동일 DB를 공유하는 상태 | 설계는 이 구조를 기반으로 개선 방향을 제시 |

## 1.4.5. 주요 고려사항

### 권한 정합성 vs 성능

캐시를 통한 응답 성능 향상과 권한 데이터의 실시간 정합성 사이에는 상충관계가 존재한다. 권한 변경 이벤트를 즉시 캐시에 반영하는 설계를 기본으로 하되, 이벤트 발행 실패 시의 복구 방안도 함께 고려한다.

### 분산 환경에서의 로컬 캐시 일관성

다수의 포털 서버 인스턴스가 각각 로컬 캐시(Caffeine)를 보유하는 구조에서는 인스턴스 간 캐시 불일치가 발생할 수 있다. 이를 허용 가능한 수준으로 관리하기 위한 TTL 정책과 캐시 무효화 전략을 설계에 포함한다.

### 커넥션 풀 분리에 따른 총 커넥션 수 증가

기능별로 커넥션 풀을 분리하면 동일한 DB에 연결되는 총 커넥션 수가 증가한다. DB 서버의 최대 커넥션 수 제한을 고려하여 각 풀의 크기를 적절히 설정해야 한다.
