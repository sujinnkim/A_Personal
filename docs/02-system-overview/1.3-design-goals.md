# 1.3. 설계 목표

본 장에서 식별한 이슈(→ 1.2 이슈 참조)를 해소하기 위한 아키텍처 개선의 설계 목표를 정의한다. 각 목표는 측정 가능한 지표와 연결되며, 품질 요구사항으로 구체화된다.

---

## 1.3.1. 피크 시 권한 갱신 응답시간 단축

**목표:** 로그인 시 권한 갱신을 포함한 회원 정보 조회 응답 시간을 피크 시간대에도 평균 1초 이내로 유지한다.

**배경:** 현행 `GET /members/{email}` API는 AC서버, LLM 권한, 용어사전 권한 등 다수의 외부 서버에 대해 매 로그인마다 동기적 갱신을 수행한다. 권한 데이터는 변경 빈도가 낮음에도 불구하고 매 호출마다 외부 서버를 거치는 구조로 인해 응답 지연이 발생한다 (→ ISSUE-01).

**달성 방향:** 권한 데이터를 캐시에 저장하여 평상시 캐시에서 즉시 응답하고, 권한 변경 시에만 캐시를 갱신하는 구조로 전환한다.

---

## 1.3.2. 2만 명 동시 입장 안정 처리

**목표:** 2만 명이 동시에 입장 요청을 전송하는 단발성 burst 상황에서 DB 커넥션 풀 사용률을 80% 이하로 유지하고, 평균 응답 시간 1초 이내를 보장한다.

**배경:** 스트리밍 서비스 시작 시점에 대규모 동시 입장 요청이 집중되며, WC서버 동기 호출로 인한 스레드 점유가 스레드 고갈 및 DB 커넥션 고갈로 이어지는 위험이 존재한다 (→ ISSUE-02).

**달성 방향:** 논블로킹 처리 방식을 도입하여 스레드 점유를 최소화하고, 입장 시 재사용 가능한 공통 정보를 단기 캐시하여 DB 조회 횟수를 줄인다.

---

## 1.3.3. 특정 서비스 커넥션 고갈 시 타 서비스 정상 운영 보장

**목표:** 입장 처리 서비스의 DB 커넥션 풀이 고갈되더라도, 회의 조회 및 회의 개설 등 다른 서비스는 정상적으로 DB에 접근하여 서비스를 제공할 수 있어야 한다.

**배경:** 현재 DB 커넥션 풀이 기능 간 격리 없이 공유되어 있어, 특정 기능의 burst가 전체 서비스 장애로 전파되는 구조다 (→ ISSUE-03).

**달성 방향:** 기능 단위(입장/조회/기타)로 DB 커넥션 풀을 분리하여 벌크헤드(Bulkhead) 패턴을 적용한다. 각 풀은 독립적인 최대 커넥션 수와 획득 타임아웃을 갖는다.

---

## 1.3.4. 핵심 기능 성공률 99.9% 이상 유지

**목표:** 트래픽 집중 구간에서 회의 입장, 회의 조회, 회의 개설 등 핵심 기능의 요청 성공률이 99.9% 이상을 유지한다.

**배경:** 커넥션 풀 고갈, 스레드 고갈, 외부 서버 연쇄 지연 등의 복합적 원인으로 피크 시간대 성공률 저하 위험이 존재한다 (→ ISSUE-02, ISSUE-03, ISSUE-04).

**달성 방향:** 벌크헤드 패턴으로 장애를 격리하고, 논블로킹 처리로 처리량을 향상하며, 외부 서버별 타임아웃 세분화와 fallback 메커니즘으로 연쇄 장애를 차단한다.
