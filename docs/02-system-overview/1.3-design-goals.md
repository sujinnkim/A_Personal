## DG-01. 피크 시 권한 갱신 응답시간 단축

**해결 이슈:** ISSUE-02 (로그인 시 권한 갱신 외부 서버 종속과 서버사이드 캐시 부재)

**목표:** 로그인 시 권한 갱신을 포함한 회원 정보 조회 응답 시간을 피크 시간대에도 평균 1초 이내로 유지한다.

**배경:** 현행 `GET /members/{email}` API는 AC서버·Copilot Admin 서버를 매 로그인마다 호출하고 `CompletableFuture.allOf()` 대기 구조로 모든 결과를 동기 수집한다. 권한 데이터는 변경 빈도가 낮음에도 불구하고 매 호출마다 외부 서버를 거치는 구조로 인해 응답 지연이 발생한다.

**달성 기준:** 피크 시간대 `GET /members/{email}` API 평균 응답시간 **1초 이내** (→ QA-01)

---

## DG-02. 2만 명 동시 입장 안정 처리

**해결 이슈:** ISSUE-01 (2만 명 동시 입장 시 스레드·커넥션 풀 고갈)

**목표:** 2만 명이 동시에 입장 요청을 전송하는 트래픽 집중 상황에서 DB 커넥션 풀 사용률을 80% 이하로 유지하고, 평균 응답 시간 1초 이내를 보장한다.

**배경:** 스트리밍 서비스 시작 시점에 대규모 동시 입장 요청이 집중되며, Feign 동기 호출 read timeout 3,000ms로 인한 스레드 점유가 스레드 고갈 및 DB 커넥션 고갈로 이어지는 위험이 존재한다.

**달성 기준:** 2만 명 동시 입장 시 DB 커넥션 풀(입장 전용) 사용률 **80% 이하**, 평균 응답시간 **1초 이내** (→ QA-02)

---

## DG-03. 특정 기능 커넥션 고갈 시 타 기능 정상 운영 보장

**해결 이슈:** ISSUE-04 (회의 입장 집중 시 단일 커넥션 풀 고갈의 전체 서비스 장애 전파)

**목표:** 입장 처리 기능의 DB 커넥션 풀이 고갈되더라도, 회의 조회 및 회의 개설 등 다른 기능은 정상적으로 DB에 접근하여 서비스를 제공할 수 있어야 한다.

**배경:** front-api·server-api 각 인스턴스 내 단일 HikariCP 풀을 모든 기능이 공유하며, 동일 DB를 공유하는 구조상 특정 기능의 트래픽 집중이 전체 서비스 장애로 전파된다.

**달성 기준:** 입장 전용 커넥션 풀 고갈 상황에서 회의 조회 API 응답 성공률 **100%** 유지 (→ QA-03)

---

## DG-04. 핵심 기능 성공률 99.9% 이상 유지

**해결 이슈:** ISSUE-01, ISSUE-04, ISSUE-05, ISSUE-06, ISSUE-07

**목표:** 트래픽 집중 구간에서 회의 입장, 회의 조회, 회의 개설 등 핵심 기능의 요청 성공률이 99.9% 이상을 유지한다.

**배경:** 커넥션 풀 고갈, 스레드 고갈, 외부 서버 연쇄 지연, DB lock 경합 등의 복합적 원인으로 피크 시간대 성공률 저하 위험이 존재한다. 특히 사용자 입장 요청과 cPaaS 피드백이 동시에 유입되면 참석자 상태 테이블에 DB lock 경합이 발생한다.

**달성 기준:** 트래픽 집중 구간 핵심 기능(회의 입장·조회·개설) 요청 성공률 **99.9% 이상** (→ QA-04)

---

## DG-05. 요청 유형별 처리 우선순위 제어

**해결 이슈:** ISSUE-03 (회의 입장 요청과 단순 조회의 처리 우선순위 미분리)

**목표:** 요청 집중 구간에 conference-token 발급·입장 파라미터 생성 등 핵심 입장 처리가 단순 조회·권한 갱신보다 우선 처리되도록 요청 유형별 처리 순서를 제어한다. 중요도가 낮은 요청이 스레드를 선점하여 핵심 입장 처리를 지연시키는 역전 현상을 방지한다.

**배경:** 현행 서블릿 스레드 풀은 모든 API 요청을 단일 FIFO 큐로 처리하여, 피크 시간대에 단순 조회가 스레드를 선점하면 가장 중요한 회의 입장 처리가 큐에서 대기하는 역전이 발생한다.

**달성 기준:** 요청 집중 구간에서 회의 입장 요청의 처리 지연이 단순 조회 요청에 의해 유발되지 않음 — 입장 처리 응답시간이 단순 조회 유입 증가에 무관하게 **1초 이내** 유지 (→ QA-02)

---

## DG-06. 예측 가능한 피크 구간 선제 대응

**해결 이슈:** ISSUE-09 (예약 회의 기반 트래픽 예측에도 선제 대응 구조 부재)

**목표:** 예약 회의 데이터로 피크 시점을 사전에 인지하여 집중 구간 진입 전에 캐시·커넥션을 워밍함으로써, cold start 없이 피크를 맞이한다. 현재는 트래픽이 실제 집중된 후에야 반응하는 구조로, 가장 많은 사용자가 몰리는 순간이 동시에 시스템 응답이 가장 느린 순간이 된다.

**배경:** 예약 회의 데이터와 참석자 수가 DB에 존재함에도 이를 활용한 사전 대응이 없어, 피크 집중 구간 초입에 캐시 미스·커넥션 cold start 상태에서 요청을 처리하게 된다.

**달성 기준:** 피크 집중 구간 진입 시점부터 핵심 입장 경로의 캐시 히트율 **목표 수준 유지** — cold start로 인한 응답 지연 스파이크 없이 피크 초입부터 QA-02 기준 충족 (→ QA-02, QA-04)

---

## DG-07. 외부 연계 의존성 격리 및 독립 제어

**해결 이슈:** ISSUE-08 (외부 연계 시스템 증가에 따른 단일 코드베이스 의존성 누적)

**목표:** 외부 연계 시스템 증가에 따른 단일 코드베이스 복잡도를 줄이고, 연계별로 독립적인 타임아웃·재시도·폴백 정책을 적용하며 피크 대응 시 특정 연계만 선별적으로 격리·제어할 수 있는 구조를 만든다.

**배경:** 현재 10개 이상의 외부 연계 로직이 단일 코드베이스에 혼재하여, 피크 시 특정 연계만 차단하거나 정책을 긴급 조정하려면 yml 수정 후 어플리케이션 재구동이 필요하여 트래픽 집중 상황에서의 런타임 긴급 대응이 불가능하다. 연계 수 증가에 비례해 피크 대응 전략의 실효성이 저하되는 구조다.

**달성 기준:** 신규 외부 연계 추가 또는 연계별 정책(타임아웃·폴백) 변경 시 **다른 연계 및 포털 서비스 재배포 없이** 적용 가능 (→ QA-04)
