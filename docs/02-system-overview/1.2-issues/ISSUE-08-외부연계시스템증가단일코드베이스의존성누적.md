# ISSUE-08. 외부 연계 시스템 증가에 따른 단일 코드베이스 의존성 누적

## 현황

포털 서버는 10개 지역의 WC서버, 3개 지역의 VC서버, AC서버를 비롯해 신규 도입된 AI서버, Copilot Admin 서버, 연계시스템 A·B·C 등 10개 이상의 외부 시스템과 통합 운영된다. 서비스 확대(대외 고객사, 공공기관, 스트리밍 서비스)에 따라 연계 대상은 계속 증가하는 추세이며, 모든 연계 로직이 단일 코드베이스에 집중되어 있다.

## 문제점

- 신규 외부 시스템 연계 추가 시 포털 서버 전체 코드를 수정하고 어플리케이션 재구동이 필요하다.
- 연계 시스템별 인증 방식, 타임아웃 정책, 재시도 로직은 Hystrix Feign 설정으로 개별 적용되어 있으나, 모두 단일 코드베이스에 산재해 있어 특정 연계 설정만 긴급 조정하려 해도 yml 수정 후 어플리케이션 재구동이 강제된다.
- 단일 코드베이스 내에 모든 연계가 집중되어 있어, 연계 수가 늘어날수록 변경·운영·장애 원인 파악의 복잡도가 증가한다.

## 요청 집중 구간에서의 심화

요청 집중 구간에는 다수의 외부 서버 호출이 동시에 폭발적으로 발생한다. 이 상황에서 모든 연계 로직이 단일 코드베이스에 집중되어 있으면 다음 문제가 심화된다.

- **선별적 격리 불가**: 연계별 타임아웃·재시도 정책은 Hystrix Feign 설정으로 개별 적용되어 있으나, 단일 코드베이스에 모여 있어 특정 연계 설정을 긴급 조정하려면 yml 수정 후 어플리케이션 재구동을 거쳐야 한다. 피크 대응 중 재구동 없이 런타임에서 즉각적인 정책 조정이 불가능하다.
- **연계별 독립 확장 불가**: 특정 연계 경로에 트래픽이 집중될 때 해당 연계 처리만 선택적으로 수평 확장하거나 리소스를 재배분하는 것이 불가능하다.
- **장애 원인 파악 지연**: 피크 구간에 장애가 발생했을 때 10개 이상의 연계가 동일 코드베이스에 혼재되어 있으면 원인이 되는 연계 시스템을 신속히 특정하기 어렵다.

## 영향

- 신규 연계 추가·변경 시마다 어플리케이션 재구동이 강제되어 서비스 가용성에 영향
- 연계 시스템 증가에 따른 코드 복잡도 증가로 개발·운영 효율 저하
- 개별 연계 시스템의 독립적 확장·교체가 불가능하여 서비스 진화 속도 저하
- 요청 집중 구간 대응을 위한 연계별 세밀한 정책 조정(타임아웃·격리·스로틀링)이 불가능하여 피크 대응 전략의 실효성 저하
