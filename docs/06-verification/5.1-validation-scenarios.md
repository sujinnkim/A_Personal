# 5.1. 아키텍처 검증 시나리오

## 5.1.1 검증 목표

본 절은 4장에서 설계한 아키텍처 전략(AS)이 *요청 집중 구간의 병목을 실제로 해소하는가*를 프로토타입에서 정량 검증하는 시나리오를 정의한다. 검증의 초점은 각 전략의 *적용 전후 결과 속성*(응답시간·풀 격리·성공률·라우팅 정합)이다. 8만 명을 물리적으로 재현하는 대신 *규모에 비종속인 메커니즘의 성질*을 축소 환경에서 확인한다(C-06). 각 시나리오는 상위 품질 요구사항(QA)과 1:1 또는 다대일로 연결된다.

## 5.1.2 검증 범위: 직접 검증 vs 컨셉 검증

검증 깊이는 *프로토타입에서 정량 실측이 가능한가*로 갈리며, 그 분류는 [4.1.3 구현·검증 대상](../05-architecture-design/4.1-conceptual/4.1.3-implementation-scope.md)을 따른다.

| 검증 유형 | 대상 전략 | 방법 | 시나리오 |
|---|---|---|---|
| ★ **직접 검증 (부하 측정)** | AS-02·AS-03·AS-04·AS-05·AS-08 | k6 부하로 응답시간·풀 사용률·성공률을 적용 전후 정량 비교 | IV-01·IV-02·IV-03 |
| ★ **직접 검증 (사건 주입·라우팅 집계)** | AS-09·AS-07 | stub 장애 주입(IV-04) 또는 write/read 경로 분리 집계(IV-05)로 결과를 정량 판정 | IV-04·IV-05 |
| ○ **컨셉 검증 (전용 부하 시나리오 비대상)** | AS-01·AS-06 | 구조·동작 재현으로 결과 속성만 확인 | 본 절 비대상 |

> **컨셉 검증 전략(비대상).** **AS-01**(도메인 경계 분리)은 부하·주입 측정 대상이 아니라 패키지 경계·ArchUnit(`DomainBoundaryArchTest`)·전용 DataSource라는 *구조 재현*으로 확인한다. **AS-06**(피크 Throttling)은 프로토타입에 구현되어 있으나(`ThrottlingInterceptor`·`PeakDetector`, 4.2.2 참조) DB 커넥션 병목과 직접 관련이 없는 별도 유입 제어 관심사라 본 절의 전용 부하 시나리오 대상에는 포함하지 않으며, 설계 논거는 [4.3.2 요구 분석](../05-architecture-design/4.3-evaluation/4.3.2-requirements-analysis.md)에서 다룬다.

> **검증 환경의 측정 한계.** 8만 명 동시 입장은 단일 서버 축소 시뮬레이션으로 대체한다(C-06). 절대 처리량이 아니라 *적용 전후 거동 차이*를 판정 근거로 삼으며, 각 한계는 해당 시나리오에 명시한다.

## 5.1.3 검증 환경

검증은 운영 환경을 *동등 패턴*으로 축소한 docker-compose PoC 슬라이스에서 수행한다([guides](../../guides/) 참조). 운영의 GKE·Cloud SQL·Cloud Memorystore 대신 동일 제품군 컨테이너를 사용한다.

### 구성 요약

| 구분 | PoC 구성 | 운영 대응 |
|---|---|---|
| 컴퓨트 | 단일 호스트 docker-compose. `front-api` 컨테이너 | GKE front-api Deployment × N (HPA) |
| 라우팅 | `nginx`(8081/8080 URL 패턴 분리) | Cloud HTTPS LB + Nginx Ingress |
| DB | `mariadb-primary` + `mariadb-replica`(복제) | Cloud SQL for MariaDB (Primary + Read Replica) |
| 캐시 | `redis` + front-api 내 L1 Caffeine | Cloud Memorystore + L1 |
| 외부 서버 | `stub-server`(정상/장애·지연 주입) | Meeting Manager·AC·Copilot |
| 관측 | `prometheus` + `grafana` | 상용 관측 |
| 부하 생성기 | **k6**(`iv01~iv05`) | (검증 전용) |

### 운영 → 검증 축소

운영의 다중 인스턴스·8만 동시 입장을, 단일 front-api 컨테이너 + k6 축소 부하(수백 TPS·수백 VU)로 축소했다. 검증 핵심인 *비동기 스레드 반환·기능별 풀 격리·CQRS 라우팅·CB 차단·캐시 완충*은 인스턴스 수나 부하 규모에 비례하지 않는 *구조적 성질*이다. 축소된 부하에서도 적용 전후 비교로 동일하게 성립한다.

### 외부 서버 → stub-server 대체

외부 벤더 서버(Meeting Manager·AC·Copilot)는 모킹 대상이므로(C-05) `stub-server` 컨테이너로 대체한다. stub은 정상 응답과 함께 **장애·지연 주입**(`POST/DELETE /stub/{server}/fault`, `delayMs` 지정 시 지연)을 지원한다. 이것으로 AS-09 Circuit Breaker 동작(IV-04)을 운영 외부 서버 없이 재현한다. MariaDB·Redis는 운영과 동일 오픈소스 제품이라 별도 대체가 없다.

## 5.1.4 개별 시나리오

### 5.1.4.1 IV-01. AS-02·04·08: 동시 입장 집중 시 풀 사용률·응답시간 보전

**해소 문제:** ISSUE-01·03·04 (동기 병목·우선순위 역전·풀 고갈) · **관련:** QA-02 (AD-02)

> 8만 명 규모 동시 입장 집중 구간에서 커넥션 풀 사용률이 80% 이하로 유지되고 평균 응답 1초 이내인가.

- **검증 접근법**: 입장(`/meetings/{id}/join`, 8081) 요청을 피크까지 점증시키며, AS-02(@Async 서블릿 스레드 즉시 반환)·AS-04(입장 전용 Connector)·AS-08(join-pool Bulkhead)이 함께 작동할 때의 응답시간·풀 사용률을 관측한다.
- **부하 검증 절차** (k6 `iv01-join-load.js`)
  1. ramping-vus: 1분 100 → 2분 500(피크) → 3분 500 유지 → 1분 50 → 30초 0.
  2. 피크·유지 구간의 입장 성공률·응답시간(p50·p95)과 HikariCP `join-pool` activeConnections를 수집한다.
- **관측 지표**

| 지표 | 소스 | 합격 기준 |
|---|---|---|
| 입장 성공률 | k6 `join_success_rate` | > 95% |
| 입장 응답시간 | k6 `join_duration_ms` | p50 < 1,000ms · p95 < 3,000ms |
| HTTP 실패율 | k6 `http_req_failed` | < 5% |
| join-pool 사용률 | HikariCP/Prometheus | ≤ 80% |

- **예상 결과**: 서블릿 스레드가 외부 응답을 기다리지 않고 즉시 반환(AS-02)되므로 피크에도 스레드 풀이 고갈되지 않는다. 입장 전용 8081 경로(AS-04)는 단순 조회와 스레드 경합을 차단하고 join-pool(AS-08)은 다른 풀과 격리된다. 풀 사용률은 80% 이하, 응답은 1초 이내로 유지된다.

### 5.1.4.2 IV-02. AS-08: join-pool 고갈 시 service-pool 격리

**해소 문제:** ISSUE-04 (풀 공유로 인한 연쇄 장애) · **관련:** QA-03 (AD-03)

> 입장 풀(join-pool)이 포화된 상태에서 회의 조회 등 타 기능(service/query-pool)의 성공률이 100% 유지되는가.

- **검증 접근법**: join-pool(100)을 초과하는 부하로 의도적 포화·대기를 유발한 채, 별도 풀을 쓰는 조회 API를 동시 호출해 격리 여부를 본다.
- **부하 검증 절차** (k6 `iv02-pool-isolation.js`)
  1. 시나리오1: 입장 VU 120(join-pool 100 초과) 3분 → join-pool 포화·대기.
  2. 시나리오2: 포화 30초 뒤 조회 API 3분 동시 호출.
- **관측 지표**

| 지표 | 소스 | 합격 기준 |
|---|---|---|
| 조회 성공률 | k6 `query_success_rate` | > 99.9% |
| 조회 응답시간 | k6 `query_duration_ms` | p95 < 500ms |
| 조회 HTTP 실패율 | k6 `http_req_failed{query_isolation}` | < 0.1% |

- **예상 결과**: join-pool이 고갈되어 입장이 실패하더라도 service/query-pool은 독립 DataSource라 조회 API 성공률 100%·응답을 유지한다(물리 격리, QA-03 충족). 단일 풀이라면 조회도 함께 실패하지만 풀을 분리해 이를 막는다.

### 5.1.4.3 IV-03. AS-03·05: 캐시 기반 권한 갱신 응답 보전

**해소 문제:** ISSUE-02·09 (외부 권한 조회 지연·cold start) · **관련:** QA-01 (AD-01)

> 피크 권한 갱신 집중 구간에서 L1·L2 캐시와 선제 적재로 평균 응답 1초 이내가 유지되는가.

- **검증 접근법**: Pre-warming(AS-05)으로 1,000명 권한을 L2에 선적재한 상태에서, 권한 조회(`GET /members/{email}/detail`)를 피크까지 점증시키며 캐시 hit에 의한 응답·외부 호출 회피를 관측한다.
- **부하 검증 절차** (k6 `iv03-cache-hit.js`)
  1. Pre-warming으로 1,000 이메일 L2 선적재(반복 이메일로 hit 유도).
  2. ramping-vus: 1분 200(피크 진입) → 3분 300 유지 → 1분 0.
- **관측 지표**

| 지표 | 소스 | 합격 기준 |
|---|---|---|
| 권한 조회 성공률 | k6 `auth_success_rate` | > 99% |
| 권한 조회 응답시간 | k6 `auth_duration_ms` | avg < 1,000ms · p95 < 2,000ms |
| HTTP 실패율 | k6 `http_req_failed` | < 1% |
| 캐시 hit율 | front-api/Prometheus | 높은 hit율(외부 호출 최소) |

- **예상 결과**: 선적재된 권한이 L1(Caffeine)·L2(Redis)에서 hit되어 외부 서버 호출 없이 즉시 반환된다. 피크 초입에도 cold start 없이 평균 1초 이내가 유지된다(AS-03 완충 + AS-05 선제 적재).

### 5.1.4.4 IV-04. AS-09: 외부 서버 장애 시 Circuit Breaker 차단·Fallback

**해소 문제:** ISSUE-05·06 (외부 동기 호출 연쇄 지연·차등 격리 부재) · **관련:** QA-05 (AD-05)

> 외부 서버 장애를 주입했을 때 CB가 fail-fast로 차단하고 무관한 핵심 기능의 성공률이 99.9% 이상 유지되는가.

- **검증 접근법**: `stub-server`에 서버별 장애·지연을 주입(`/stub/{server}/fault`)하고 입장·token·권한·조회 등 핵심 기능을 동시 부하하며 CB Open·fail-fast·계층 Fallback 동작을 본다.
- **부하 검증 절차** (k6 `iv04-circuit-breaker.js`, `STUB_URL` 8090)
  1. 특정 외부 서버(예: Meeting Manager 또는 Copilot)에 장애/지연 주입.
  2. 핵심 기능 4종(join·token·auth·query)을 부하하며 성공률·응답을 수집한다. 주입 해제 후 회복을 확인한다.
- **관측 지표**

| 지표 | 소스 | 합격 기준 |
|---|---|---|
| 핵심 기능 성공률(4종) | k6 `join/token/auth/query_success_rate` | > 99.9% |
| CB 상태·fail-fast | Resilience4j/Prometheus | 장애 시 Open 전환·timeout 없이 즉시 거부 |
| Fallback 동작 | 로그/응답 | AC: DB Fallback · Copilot: Redis→DB Fallback |

- **예상 결과**: 장애 서버의 CB가 임계 초과 시 Open으로 전환되어 timeout 없이 즉시 거부(fail-fast)한다. `externalCallExecutor` 스레드는 장애 호출로 소진되지 않는다. AC·Copilot은 계층 Fallback으로 서비스 연속성을 유지하므로 무관 기능 성공률은 99.9% 이상이다(AS-09 차등 CB).

### 5.1.4.5 IV-05. AS-07: write 집중 중 조회 경로 분리(CQRS)

**해소 문제:** ISSUE-07 (대규모 회의 시작 시 write/read 경합) · **관련:** QA-04 (AD-04)

> cPaaS 피드백 write가 집중되는 동안 참석자 상태 조회(read)가 Replica로 분리되어 성공률·응답이 유지되는가.

- **검증 접근법**: 참석자 상태 피드백 write(`ParticipantFeedback`, Primary/service-pool)를 집중시킨 채, 상태 조회 read(Replica/query-pool, `@Transactional(readOnly)`)를 동시 호출해 write/read 경로 분리를 본다.
- **부하 검증 절차** (k6 `iv05-participant-cqrs.js`)
  1. 시나리오1: 입장 완료·퇴장 피드백 write 집중 3분(Primary).
  2. 시나리오2: 30초 뒤 참석자 상태 조회 read 3분 동시(Replica).
- **관측 지표**

| 지표 | 소스 | 합격 기준 |
|---|---|---|
| 상태 조회 성공률 | k6 `status_read_success_rate` | > 99.9% |
| 상태 조회 응답시간 | k6 `status_read_duration_ms` | p95 < 500ms |
| 조회 HTTP 실패율 | k6 `http_req_failed{status_read_isolation}` | < 0.1% |
| 라우팅 정합 | Primary/Replica 세션 | write는 Primary, read는 Replica로 라우팅 |

- **예상 결과**: `RoutingDataSource`·`DataSourceRoutingAspect`가 readOnly 트랜잭션을 Replica로 분리한다. write 집중(Primary) 중에도 조회 read(Replica)는 경합 없이 성공률 99.9%·응답을 유지한다(AS-07 CQRS 라우팅). 결과 집계·회복 상세는 [5.2 구현 및 결과](5.2-implementation-and-results.md)에서 다룬다.
