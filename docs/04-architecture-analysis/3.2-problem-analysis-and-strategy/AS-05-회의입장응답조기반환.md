# AS-05. 회의 입장 응답 조기 반환

## 적용 대상

> **전제**: AS-02(입장 처리 경로 비동기 전환)의 파생 전략. AS-02의 `@Async` 기반 비동기 처리 기반이 없으면 이 전략은 단독 적용 불가.

- **아키텍처 드라이버**: AD-02 (2만 명 동시 입장 처리 성능), AD-04 (핵심 기능 성공률 99.9%)
- **해결 이슈**:
  - ISSUE-01: UC-04(회의 입장) 흐름에서 Meeting Manager Feign 동기 호출이 완료될 때까지 클라이언트는 응답을 받지 못한다. 이 대기 구간이 서블릿 스레드 점유의 직접 원인이다. 클라이언트 입장에서 실제로 필요한 것은 "입장 파라미터(conference-token + 파라미터 값)"이며, Meeting Manager로의 전달 완료 확인이 즉시 응답에 필수적이지 않을 수 있다.
  - ISSUE-02: `GET /members/{email}` API에서 `CompletableFuture.allOf()` 대기로 가장 느린 외부 서버가 전체 응답 시간을 결정한다. 비동기 후처리 패턴을 적용하면 가용 권한 데이터만으로 즉시 응답하고 나머지는 후처리할 수 있다.
  - ISSUE-05: server-api의 VC/AC 서버 호출은 회의 개설 결과를 즉시 확정해야 하므로 완전한 Eager Response 적용은 어렵다. 단, 응답 지연이 큰 구간(외부 서버 호출)을 AS-02 비동기로 분리하여 서블릿 스레드 반환 속도를 높인다.
- **설계 목표**: DG-02 (2만 명 동시 입장 안정 처리), DG-04 (핵심 기능 성공률 99.9%)
- **관련 유스케이스**: UC-04 (회의 입장)
- **관련 품질 요구사항**: QA-02 (동시 입장 처리 성능), QA-04 (핵심 기능 가용성)

## 설계 근거

UC-04(회의 입장)의 현행 처리 흐름을 단계별로 분해하면 다음과 같다:

```
① DB에서 입장 가능 여부 확인         (포털 서버 내부, 빠름)
② conference-token 발급              (포털 서버 내부, 빠름)
③ 입장 파라미터 생성                  (포털 서버 내부, 빠름)
④ Meeting Manager에 입장 파라미터 전달   (Feign 동기, 최대 3,000ms ← 병목)
⑤ 클라이언트에 입장 파라미터 반환
```

①~③은 포털 서버 내부에서 처리되어 빠르다. ④에서 Meeting Manager Feign 동기 호출이 완료되어야 ⑤를 반환할 수 있는 구조가 병목이다. 2만 명 동시 입장 시 2만 개의 서블릿 스레드가 ④ 단계에서 최대 3,000ms씩 블로킹된다.

여기서 핵심 질문은 **"⑤를 반환하기 위해 ④가 반드시 완료되어야 하는가?"**이다. 클라이언트(PC 클라이언트, 모바일)가 회의에 실제 접속하려면 conference-token과 입장 파라미터 값이 필요하다. Meeting Manager가 해당 파라미터를 수신하여 cPaaS에 세션을 등록하는 것은 포털 서버의 내부 처리 단계이지, 클라이언트가 즉시 필요로 하는 정보가 아니다. Meeting Manager 전달이 미완료 상태에서 PC 클라이언트가 입장을 시도하면 cPaaS 세션이 아직 준비되지 않은 상태일 수 있으나, 이 간격을 재시도 메커니즘으로 흡수할 수 있다.

이 판단을 기반으로 **③ 완료 후 즉시 클라이언트에 응답을 반환**하고, ④는 AS-02의 비동기 큐를 통해 후처리하는 설계가 가능하다.

## 대안

### 대안 1. 현행 동기 응답 (Meeting Manager 응답 완료 후 반환)

**개념**: 현행 흐름 유지. ④ Meeting Manager Feign 호출 완료 후 ⑤ 클라이언트 반환.

**이 시스템 적용 방식**: UC-04 기존 처리 흐름 그대로.

**한계**: 2만 건 동시 입장 시 2만 개 서블릿 스레드가 Meeting Manager 응답(최대 3,000ms)을 대기한다. QA-02(동시 입장 처리 성능) 달성이 Meeting Manager 응답 속도에 전적으로 종속된다.

---

### 대안 2. 폴링 방식 (처리 ID 즉시 반환 + 클라이언트 폴링)

**개념**: 입장 요청 수신 즉시 처리 ID를 반환하고, 클라이언트가 처리 완료 여부를 주기적으로 폴링한다.

**이 시스템 적용 방식**: `POST /meetings/{id}/join` 응답으로 `{ "requestId": "xxx", "status": "processing" }` 반환. 클라이언트가 `GET /meetings/{id}/join/status/{requestId}`를 폴링하여 완료 여부 확인.

**한계**: 기존 클라이언트(PC 클라이언트, 모바일 앱)의 API 호출 방식 변경이 필요하다. 이는 C-02(API 하위 호환성 유지) 제약을 위반한다. 클라이언트 배포 없이 서버 측만 변경해야 한다는 제약에서 이 대안은 선택 불가다.

---

### 대안 3. 즉시 응답 + 비동기 후처리 (현행 API 스키마 유지)

**개념**: ①~③을 완료한 즉시 클라이언트에 입장 파라미터를 반환하고, ④ Meeting Manager 전달은 AS-02의 `AsyncTaskExecutor` 스레드 풀에서 비동기 후처리한다. 클라이언트는 기존 API 스키마로 동일하게 응답을 수신한다.

**이 시스템 적용 방식**:
- ①~③ 완료 후, 서블릿 스레드에서 클라이언트에 입장 파라미터 응답 반환
- ④ Meeting Manager 전달은 `@Async("externalCallExecutor")`로 비동기 실행 (서블릿 스레드와 분리)
- Meeting Manager 전달 실패 시 지수 백오프(Exponential Backoff) 재시도 큐에 등록
- Meeting Manager 전달 최종 실패 시 AS-10 Circuit Breaker fallback 처리 (conference-token 무효화 + 사용자에게 재시도 안내)
- 클라이언트 측에서는 API 응답 스키마 변경 없음 → C-02 준수

**장점**: 서블릿 스레드가 Meeting Manager 응답을 기다리지 않으므로 스레드 고갈 위험이 대폭 감소한다. 2만 건 동시 입장 시에도 서블릿 스레드는 ①~③(포털 내부 처리) 구간만 점유하고 즉시 반환된다. 클라이언트 변경 없이 서버 측만으로 적용 가능(C-02 준수).

## 채택

**채택 대안**: 대안 3 — 즉시 응답 + 비동기 후처리

**채택 근거**: 대안 1은 QA-02(동시 입장 처리 성능) 달성이 Meeting Manager 응답 속도에 종속되어 2만 명 규모에서 구조적 한계를 가진다. 대안 2는 C-02(API 하위 호환성) 위반으로 선택 불가. 대안 3은 기존 API 스키마를 유지하면서 서블릿 스레드 점유 시간을 외부 서버 호출 시간(최대 3,000ms)에서 포털 내부 처리 시간(수십~수백ms)으로 단축한다.

**적용 방향**:
- `MeetingJoinService.join()` 메서드에서 ①~③ 완료 직후 응답 객체 구성 및 반환
- Meeting Manager 전달 로직을 `@Async("externalCallExecutor")`가 붙은 별도 메서드로 추출
- `CompletableFuture`로 비동기 결과를 추적하여 실패 시 재시도 큐(Spring Retry + `@Retryable`) 연동
- Meeting Manager 전달 재시도 최대 3회, 최종 실패 시 conference-token 무효화 처리 및 모니터링 알림

**주의사항**: Meeting Manager 전달이 비동기이므로 클라이언트가 입장 파라미터를 수신한 직후 PC 클라이언트가 입장을 시도하면 Meeting Manager가 cPaaS에 세션을 아직 등록하지 않은 상태일 수 있다. 클라이언트의 입장 재시도 로직(기존 클라이언트에 이미 존재)이 이 간격을 흡수하는지 확인 필요.
