# 4.2.3.3. 데이터 계층 (캐시·DB·커넥션 풀·관측)

캐시·데이터베이스는 GCP 관리형 서비스로 배치해 이중화·백업·failover를 매니지드가 담당한다. 본 절은 4.3.4 RK-07(공유 인프라 단일 장애점)이 배치 뷰로 위임한 이중화 책임과 RK-01(커넥션 총량)을 구체화한다. 검증 환경은 동일 구조를 docker-compose로 재현한다.

## 캐시: Cloud Memorystore for Redis (AS-03)

| 계층 | 배치 | 이중화 |
|---|---|---|
| L1 | Caffeine (front-api Pod 로컬, TTL 5분) | Pod별 독립 (SPOF 아님) |
| L2 | Cloud Memorystore for Redis, Standard tier (AC 1시간 / LLM·용어사전 30분) | Standard tier = 자동 failover(primary/replica), Private Service Access |

L2는 다수 Pod가 공유하는 단일 지점이므로 Standard tier(HA)로 이중화한다. Memorystore 장애·miss 시 AS-09 계층 Fallback(외부/DB)으로 연속성을 유지한다(RK-07).

## DB: Cloud SQL for MariaDB (AS-07)

| 노드 | 역할 | 라우팅 | 이중화·복제 |
|---|---|---|---|
| Primary | Write | `@Transactional(readOnly=false)` | zone-a. 자동 백업·PITR |
| Standby | HA 대기(failover) | (평시 미사용) | zone-b. Primary와 동기 복제, 장애 시 자동 승격 |
| Read Replica ×2 | Read | `@Transactional(readOnly=true)` → `RoutingDataSource` | Primary와 비동기 복제, 읽기 확장 |

`RoutingDataSource`·`DataSourceRoutingAspect`가 readOnly 트랜잭션을 Read Replica로 라우팅한다(AS-07). Replica 장애 시 헬스체크로 Primary Fallback을 허용하고, 강정합 필요 조회는 Primary를 유지한다(RK-04·07).

## HikariCP 커넥션 풀·총량 (AS-08)

| 풀 | maximumPoolSize | connectionTimeout | 대상 |
|---|:---:|:---:|---|
| join-pool | 100 | 3,000ms | 입장(Primary) |
| service-pool | 40 | 5,000ms | 회의 시작·초대·피드백(Primary) |
| general-pool | 60 | 5,000ms | 권한·일반 조회(Primary) |
| query-pool | 80 | 3,000ms | Read(Replica) |

Pod당 Primary 커넥션 합 200(100+40+60), Replica 80이다. front-api Pod가 N개로 확장되면 200×N이 Cloud SQL 최대 커넥션을 넘지 않도록 Pod 수 상한과 함께 설계한다. 총량이 상한에 근접하면 확대 경로로 프록시(ProxySQL 등)를 도입해 DB 수신 커넥션을 Pod 수와 분리한다(RK-01).

## 관측 (Prometheus·Grafana)

`observability` 네임스페이스에 Prometheus(메트릭 수집)·Grafana(대시보드)를 배치한다. front-api가 노출하는 지표(풀 사용률·CB 상태·캐시 hit율·응답시간)를 수집해 QA 임계·RK 지표를 모니터링한다(prototype `monitoring/prometheus.yml` 기준).

| 항목 | 베이스라인 |
|---|---|
| 메트릭 수집 주기 | ≤ 15초 |
| 임계 초과 알림 | ≤ 1분 |
| 관측 인프라 가용성 | ≥ 99% |

## 운영 ↔ 검증 환경 동등성

운영은 GKE + Cloud SQL(Primary/Replica) + Memorystore이고, 검증은 동일 구조를 docker-compose(`mariadb-primary`·`mariadb-replica`·`redis`·`nginx`·`prometheus`·`grafana`·`stub-server`)로 재현한다. 풀 분리·CQRS 라우팅·CB·캐시 등 *메커니즘의 결과 속성*은 규모·환경에 무관하므로 검증 환경에서 확인한 속성이 운영에 일반화된다(C-06).
