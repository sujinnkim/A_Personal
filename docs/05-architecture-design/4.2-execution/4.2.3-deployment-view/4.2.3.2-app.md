# 4.2.3.2. GKE 클러스터 배치 (애플리케이션)

front-api·server-api·admin-api는 컨테이너 이미지로 빌드되어 GKE(Google Kubernetes Engine)에 배포된다. 이미지는 GAR(Google Artifact Registry)에 올리고 ArgoCD가 GitOps로 클러스터에 동기화한다.

## 클러스터 사양

| 항목 | 결정 | 근거 |
|---|---|---|
| 클러스터 유형 | Regional (다중 zone), private cluster | 단일 리전 다중 zone HA, 노드 외부 노출 차단 |
| 노드풀 | `app-pool`(일반 워크로드) 1개. IO 특성이 다른 워크로드가 늘면 분리 | 우리 시스템은 단일 front-api 계열이라 A_Team의 IO 노드풀 분리까지는 불필요 |
| 오토스케일 | Cluster Autoscaler + 노드 on-demand | 피크 시 Pod 증가에 맞춰 노드 확장 |
| 배포 | GAR 이미지 + ArgoCD GitOps | guide 05·06 기준 |

## 네임스페이스 설계

역할별 배포 단위를 네임스페이스로 분리해 RBAC·NetworkPolicy 경계를 준다.

| 네임스페이스 | 배포 | 비고 |
|---|---|---|
| `edge` | Nginx Ingress | 8081/8080 라우팅(AS-04) |
| `front` | front-api Deployment | 사용자 요청 처리, 트래픽 집중 대상 |
| `server` | server-api Deployment | VC/AC 벤더 연동·cPaaS 피드백 |
| `admin` | admin-api Deployment | 관리자 API |
| `observability` | Prometheus · Grafana | 관측 |

## Deployment · 수평 확장 (HPA)

| 서비스 | Pod(기준) | HPA 정책 | 관련 AS |
|---|:---:|---|:---:|
| front-api | 3 | CPU 70% 또는 요청률 기준 3→N 확장 | AS-01·02·04 |
| server-api | 2 | CPU 기준 | — |
| admin-api | 1 | 고정(트래픽 낮음) | — |

front-api는 트래픽 집중 대상이라 HPA로 수평 확장한다. L1 Caffeine 캐시는 Pod 로컬이므로 Pod가 늘어도 캐시는 공유되지 않고 L2(Cloud Memorystore)로 정합을 맞춘다(AS-03). Pre-warming 스케줄러(AS-05)는 다중 Pod 중복 실행을 막기 위해 리더 선출 또는 분산 락으로 단일 Pod만 수행하게 한다.

## 자원 한도

| 항목 | 정책 |
|---|---|
| requests/limits | Pod별 CPU·메모리 requests·limits 설정으로 노드 자원 경합 방지 |
| 커넥션 총량 | front-api Pod당 Hikari 풀 합(Primary 200 + Replica 80)이 Pod 수에 비례하므로, Pod 수 상한과 DB 최대 커넥션(CR-02)을 함께 설계(RK-01) |

## 인그레스 경로

`Cloud HTTPS LB(+Cloud Armor) → Nginx Ingress → front-api(8081/8080)`. Nginx는 `nginx.conf`의 upstream(`front_api_entry` = `/meetings/*/join`·`/conference-token`, `front_api_general` = 그 외)으로 입장/일반을 분리 라우팅한다(AS-04). 상세는 [4.2.3.1 네트워크·엣지](4.2.3.1-network-edge.md).
