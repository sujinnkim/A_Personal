# 4.1.2. 설계전략 적용 범위

3.2절의 설계 전략(AS-01 ~ AS-09)이 아키텍처의 어느 레이어·컴포넌트에 적용되는지를 정리한다. 개념도(→ 4.1.1)와 함께 읽으면 각 전략의 위치를 파악할 수 있다.

## 설계전략 적용 매핑

| AS | 전략명 | 적용 레이어 | 적용 컴포넌트 | 관련 드라이버 |
|----|-------|-----------|------------|------------|
| AS-01 | 도메인 모듈 분리 | 애플리케이션 | domain.entry · domain.auth · domain.meeting | AD-03, AD-04, AD-09 |
| AS-02 | 비동기 처리 | 애플리케이션 | AsyncTaskExecutor (외부 호출 전용 스레드 풀) | AD-02, AD-04, AD-05 |
| AS-03 | 캐시 기반 응답 속도 개선 | 캐시 | L1 Caffeine (TTL 5분) · L2 Redis (TTL 30분~1시간) | AD-01, AD-04, AD-05 |
| AS-04 | 요청 우선순위 분기 | 요청 진입 | 입장 전용 Connector (port 8081) · 일반 Connector (port 8080) | AD-02 |
| AS-06 | Time-based Throttling | 요청 진입 | Throttling Interceptor (front-api 공통 Interceptor 체인) | AD-02, AD-04 |
| AS-05 | Predictive Pre-warming | 캐시 / 인프라 | Pre-warming Scheduler · L2 Redis (AS-03 캐시 인프라 전제) | AD-04 |
| AS-07 | 경량 CQRS | DB 접근 | query-pool → MariaDB Replica · 나머지 풀 → Primary | AD-04 |
| AS-08 | HikariCP Bulkhead | DB 접근 | join-pool (100conn) · service-pool (40conn) · general-pool (60conn) · query-pool (80conn) | AD-03, AD-04, AD-05 |
| AS-09 | Circuit Breaker / Fallback | 외부 연계 | integration.meetingManager · integration.ac · integration.copilot (CB + Fallback) | AD-01, AD-04, AD-05 |

## 레이어별 전략 요약

| 레이어 | 적용 전략 | 핵심 목적 |
|-------|---------|---------|
| 요청 진입 | AS-04, AS-06 | 피크 시 입장 요청 우선 처리 · 비핵심 API 처리량 제한 |
| 애플리케이션 | AS-01, AS-02 | 도메인 경계 설정 · 외부 호출 비동기화 |
| 캐시 | AS-03, AS-05 | 외부 호출 대체 · 피크 전 선제 적재로 cold start 방지 |
| 외부 연계 | AS-09 | 외부 서버 장애 격리 · 연계별 독립 정책 적용 |
| DB 접근 | AS-07, AS-08 | 기능별 커넥션 풀 격리 · Read/Write DB 분리 |

## 설계 계층 관점

다섯 레이어에서 전략은 애플리케이션과 그 인접 계층(캐시·DB 접근)에 집중된다. 이는 계층 편중이 아니라 문제와 제약에서 나온 수렴이다. 해소 대상 이슈(스레드 풀 고갈·동기 체이닝·우선순위 역전·커넥션 풀 고갈·외부 호출 연쇄·write/read 경합·cold start)가 모두 애플리케이션·런타임 계층의 병목이고 상위 제약이 타 계층 대안을 배제하기 때문이다. C-01(Spring MVC 유지)은 런타임의 Reactive 전환을, C-03(DB 공유 유지)은 샤딩·DB 분리 같은 데이터 계층 재편을, C-04(점진적 적용)는 메시지 큐 등 새 인프라 계층 도입을 제약한다. 기각된 계층 대안의 근거는 3.2 후보 분석에 기록되어 있다.

자원·인프라 계층도 제약이 허용하는 범위에서 사용한다. AS-03은 L2 Redis 캐시 계층을, AS-07은 Read Replica를 새 자원으로 투입해 애플리케이션 전략을 뒷받침한다. 수평 확장(HPA)으로 Pod당 커넥션 합(Primary 200)이 DB 최대 커넥션(CR-02)에 근접하면 트랜잭션 프록시(ProxySQL/MaxScale)로 DB 수신 커넥션을 인스턴스 수와 분리하는 확장 경로를 배치 뷰(4.2.3, RK-01)에 둔다. 애플리케이션 계층을 중심축으로 삼되 자원 계층을 제약 내에서 활용한다. 커넥션 총량 상한은 용량 확장 결정으로 관리한다.
