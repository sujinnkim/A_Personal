# 4.3.4. 위험 요인

본 절은 각 AS 본문의 §위험 요인(R#)과 상충점(4.3.3)에서 남은 잔여 위험을 *아키텍처 수준 위험(RK)* 으로 집계한다. 영향도·발생 가능성은 *상·중·하* 3등급이며, 위험은 성격별로 ① 자원·총량, ② 비동기 스레드, ③ 정합성, ④ 공유 인프라 단일 장애점, ⑤ 운영·튜닝, ⑥ 검증 범위로 정리한다.

| ID | 위험 요인 | 영향도 | 발생 가능성 | 대응 방안 | 출처 |
|----|---------|:---:|:---:|---------|------|
| **RK-01** | 풀 정적 분할로 총 커넥션·스레드 합이 DB 서버 상한(CR-02) 초과 | 상 | 하 | 풀 크기 합을 DB 최대 커넥션 내로 설계, 풀별 사용률 메트릭 모니터링으로 배분 조정 | AS-08 R1 |
| **RK-02** | externalCallExecutor 자체가 새 고갈 지점 → 외부 지연 시 스레드 누적 | 중 | 중 | AS-09 CB로 외부 지연 조기 차단, 큐 용량·풀 사이즈 부하 시험 확정 | AS-02 R1 |
| **RK-03** | 비동기 전환 시 트랜잭션·보안 컨텍스트(ThreadLocal) 전파 누락 | 중 | 중 | TaskDecorator로 컨텍스트 명시 복사, 통합 테스트로 전파 검증 | AS-02 R2 |
| **RK-04** | 복제 지연(replication lag) → 입장 직후 조회 시 read-after-write 불일치 | 중 | 중 | 재조회 필요 경로는 `readOnly=false`로 Primary 유지, 강정합 불필요 조회만 Replica 분산 | AS-07 R1 |
| **RK-05** | 캐시 TTL 구간 stale → 권한 변경이 최대 TTL만큼 반영 지연 | 하 | 중 | 변경 빈도 낮은 권한 데이터로 적용 한정, 변경 이벤트 시 `@CacheEvict`로 L1·L2 즉시 무효화 | AS-03 R1 |
| **RK-06** | 서버별 CB 임계 오설정 → 정상 서버 과도 차단 또는 장애 감지 지연 | 중 | 중 | 서버 특성 기반 초기값(AS-09 표 56) 제공, 부하 시험·모니터링으로 임계 보정 | AS-09 R1 |
| **RK-07** | **공유 인프라 단일 장애점**: L2 Redis·Replica 장애가 다수 요청에 동시 영향 | 상 | 하 | 캐시 miss·장애 시 외부/DB Fallback(AS-09 연동), Replica 헬스체크 기반 Primary Fallback | AS-03 R2 / AS-07 R3 |
| **RK-08** | Pre-warming 스케줄러 조용한 실패 → 피크 초입 선제 적재 누락 | 중 | 하 | 워밍 실패해도 AS-03 캐시가 정상 동작, 스케줄러 상태 모니터링·알람 | AS-05 R2 |
| **RK-09** | 예약에 없는 돌발 급증 시 Throttling 미작동 → 비핵심 요청 자원 과점 | 중 | 중 | AS-04 입장 전용 경로가 핵심 입장을 1차 보호, 429는 비핵심 API로만 한정 | AS-06 R1 |
| **RK-10** | **검증 범위 한계**: 검증이 개별 병목 시나리오 중심 → 전 구간 동시 부하 통합 검증은 범위 밖 | 중 | 상 | 개별 병목 우선 검증(CR-05 구현 범위), 전 구간 통합 부하는 운영 단계 단계적 검증 | CR-05 |
| **RK-11** | **검증 규모 한계**: 8만 물리 재현 불가, 축소 시뮬레이션의 규모 일반화 가정 | 중 | 상 | 메커니즘의 규모 비종속성(비동기·풀 분리·캐시의 규모 불변 성질)을 논거화, 결과 속성으로 검증 | C-06 |

**위험 분포 (영향도 × 발생 가능성)**

| 영향도＼가능성 | 하 | 중 | 상 |
| :---: | ----- | ----- | ----- |
| **상** | RK-01, RK-07 | — | — |
| **중** | RK-08 | RK-02, RK-03, RK-04, RK-06, RK-09 | RK-10, RK-11 |
| **하** | — | RK-05 | — |

영향도 *상* 구간(RK-01·07)은 발생 가능성이 *하*이고, 발생 가능성 *상* 구간(RK-10·11)은 영향도가 *중*이다. 즉 영향·가능성이 동시에 높은 최우선 위험은 없으며, 잔여 위험은 완화 경로가 정의된 채 관리 범위에 있다.

- **공유 인프라 단일 장애점(RK-07)·자원 총량(RK-01)** 이 가장 높은 영향도를 가진다. 도메인 경계 분리(ADR-001)·기능별 Bulkhead(ADR-008)으로 서비스 내부 격리는 확보했으나, L2 Redis·Replica·기능별 풀 총량이라는 *새 공유·상한 지점*이 두드러졌다. 이는 격리 결정에 내재한 트레이드오프(TP-02)이며, 각 컴포넌트 이중화·총량 설계가 배치 뷰(4.2.3)의 책임이 된다.
- **정합성 위험(RK-04·05)** 은 캐시·Replica 도입의 대가로, 적용 범위 한정과 무효화·Primary 재조회로 관리한다.
- **검증 범위·규모 한계(RK-10·11)** 는 발생 가능성이 *상*이나 영향도는 *중*이다. 검증의 책임이 규모·환경에 무관한 *메커니즘의 결과 속성*에 한정되기 때문이다. 이 한계의 정당화는 검증 환경 설계의 핵심 논거가 된다.

전체적으로 *상* 영향도 위험(RK-01·07)은 모두 총량 설계 또는 이중화·Fallback 경로로 완화 경로가 정의되어 있으며, 미완화 잔존 위험은 본 사업 범위 밖으로 명시적으로 넘긴다.
