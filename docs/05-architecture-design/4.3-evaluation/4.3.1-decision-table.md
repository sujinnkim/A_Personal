# 4.3.1. 아키텍처 설계 결정표

3.2절에서 각 설계 전략(AS)별로 대척점이 명확한 대안을 비교해 선정한 결정을 ADR(Architecture Decision Record) 형식으로 정리한다. 각 ADR은 *채택안 · 기각 대안 · 선택 근거 · 트레이드오프(감수 비용·잔여 위험)*로 정리하며, 상세 비교는 각 AS 본문(3.2)을 참조한다. ADR 번호는 결정 의존 순서(AS 번호 순서)를 따른다.

| ID | 결정 사항 (채택안) | 기각 대안 | 선택 근거 | 트레이드오프 (감수 비용·잔여 위험) |
| :---: | ----- | ----- | ----- | ----- |
| **ADR-001** | **선별적 도메인 모듈 분리 + ArchUnit 경계 강제** (AS-01) | ① 현행 구조 유지 ② 완전 MSA 분리 | 배포·스택 변경 없이 `domain.entry` 전용 DataSource·경계 확보, AS-07·08 파생 전제 마련. C-04 준수 | ArchUnit 규칙 지속 관리(위반 시 빌드 실패), 같은 JVM 공유로 물리적 격리 제한 → 런타임 격리는 ADR-008이 보완 |
| **ADR-002** | **병목 구간만 선택적 비동기화** (@Async + externalCallExecutor) (AS-02) | ① 현행 동기 유지 ② WebFlux 전면 전환 | 코드 재작성 없이 외부 동기 호출 병목만 해소, 서블릿 스레드 즉시 반환. C-01·C-04 준수 | externalCallExecutor가 새 고갈 지점, ThreadLocal 전파·예외 처리 복잡도 → TaskDecorator·ADR-009 CB로 완화 |
| **ADR-003** | **L1(Caffeine)+L2(Redis) 계층 캐시** (AS-03) | ① 캐시 없음 ② DB 캐시 전용 | 다중 인스턴스 외부 호출 일괄 완충, ADR-005 선제 적재 기반. C-04 준수 | TTL 구간 stale(권한 반영 지연), Redis 운영·L1·L2 무효화 복잡도 → 저빈도 데이터 한정 + @CacheEvict |
| **ADR-004** | **입장 전용 Tomcat Connector(8081) 분리** (AS-04) | ① 단일 FIFO 유지 ② 인메모리 우선순위 큐 재정렬 | Tomcat 기본 기능으로 코드 변경 없이 우선순위 역전 차단, 입장 전용 스레드 예약 | 포트 라우팅 인프라(LB·방화벽) 추가, 정적 배분이라 편중 시 유휴·포화·총 스레드↑ |
| **ADR-005** | **예약 회의 데이터 기반 동적 선제 초기화** (AS-05) | ① 사후 대응 ② 고정 스케줄 워밍 | 이벤트성 피크(예약 대규모 회의)까지 인식, 외부 인프라 추가 불필요, ADR-003·006와 연동 | 예측 오차 시 워밍 낭비·외부 부하, 스케줄러 조용한 실패 → 임계값 이상 회의 한정·배치 분산, 실패 시 ADR-003 캐시로 동작 |
| **ADR-006** | **피크 예측 기반 차등 Throttling** (AS-06) | ① Throttling 없음 ② 균일 Rate Limiting | 비핵심 API만 피크 구간 선별 제한해 핵심 입장 경로 무영향, ADR-005 피크 감지 공유 | 돌발 급증 미작동, 429 UX 저하, 임계 튜닝 필요 → ADR-004 전용 경로가 최소 안전망 |
| **ADR-007** | **@Transactional(readOnly) 기반 Primary/Replica 라우팅** (AS-07) | ① 선택적 수동 라우팅 ② 이벤트 소싱 + 완전 CQRS | 기존 JPA 유지, readOnly 경계로 read를 Replica로 물리 분리해 Primary write 집중. C-04 준수 | 복제 지연 read-after-write 불일치, Replica 장애 시 read 경로 영향 → 재조회 경로 Primary 유지 |
| **ADR-008** | **기능별 HikariCP 풀 + 스레드 풀 이중 Bulkhead** (AS-08) | ① 단일 풀 크기 확대 ② 커넥션 풀만 분리 | 커넥션·스레드 동시 격리로 QA-03·QA-05 충족, ADR-002 비동기 위에서 스레드 격리 의미 확보 | 정적 분할로 총 커넥션·스레드↑ → CR-02(총량 상한) 압박 → 풀 합 상한 내 설계·사용률 모니터링 |
| **ADR-009** | **외부 서버별 차등 CB + 계층적 Fallback** (Resilience4j) (AS-09) | ① Hystrix 유지 ② Resilience4j 균일 설정 | 서버 특성별 정밀 격리, Hystrix 유지보수 중단 회피, ADR-002·003과 fallback 연동 | 서버별 CB 파라미터 튜닝 부담, Fallback DB·Redis 값 stale → 특성 기반 초기값·저정합 데이터 한정 |

각 결정의 상세 대안 비교와 채택 근거는 3.2절 해당 AS 항목에 있다. **기반(횡단) 결정 ADR-001·002·003·008·009**는 여러 품질 요구에 공통으로 작용하는 플랫폼 수준 결정이고, **파생·개별 결정 ADR-004·005·006·007**은 특정 경로에 적용된다. 결정 의존은 ADR-001 → 007·008(도메인 경계 위에서 DB 경로·풀 분리), ADR-003 → 005 → 006(캐시 인프라 위에서 선제 적재, 그 위에서 유입 제한)로 누적된다.
