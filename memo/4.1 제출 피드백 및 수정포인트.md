## 원문 피드백 (평가자 작성)

제가 Meeting의 내부 구조를 잘 몰라서 읽으면서 느낀 점을 작성하였습니다. 그래서 이해 부족으로 잘못 리뷰를 드리는 것도 있을 거라고 생각합니다. 그런 부분은 양해 부탁드립니다. 그리고 리뷰 작성은 제가 읽어 가면서 작성한 거라 문서 앞에서 설명이 안되거나 이해가 안되서 리뷰로 작성한 부분이 문서 뒤에서 설명이 되는 부분도 있을 겁니다. 그런 부분이 있다면 한편으로는 무시하셔도 되고 한편으로는 설명하는 부분을 보다 앞으로 오도록 하는 것은 어떨까 합니다.

미팅 포털 서버의 역할 테이블에서 wzProParam이 무엇인지 설명이 있으면 좋을 거 같습니다. Meeting을 내부적으로 개발하거나 잘 아시는 분들은 아시겠지만, 그것을 모르는 분들은 어떤 내용인지 이해하기 어려울 거 같습니다.

아키텍처 문제 분석 및 설계 전략
AD와 AS ID의 순서가 정리가 안된 거 같습니다. ISSUE ID 순서에 맞춰서 그런거 같은데, 그러면 ISSUE ID 부여를 우선 순위에 두지 않아서 그런 걸까요? 일부는 순서가 바뀔 수 있지만 가능하면 순서 맞게 ID를 부여하면 좀 더 정리된 듯해 보이지 않을까 싶습니다.

개인적으로는 프로님이 ID를 임의로 부여하고 우선순위에 맞게 매칭하는 것이 괜찮다고는 생각하는데 평가하시는 분들은 다른 기준을 가지고 계실 수도 있지 않을까 해서요.

두 개의 Issue가 동일한 AD를 공유합니다. 이런 Issue들끼리는 어쩌면 강하게 연결되어 있어서 어쩌면 하나의 Issue를 나눈게 아닌가 싶습니다. 즉 원인은 하나인데 나타나는 증상으로 나눈 것일 수 있을 거 같습니다. 제가 정확하게 몰라서 그럴 수 있습니다만, 한번 더 검토해 보시고 Issue를 합치거나 AD를 나누는 것이 좋지 않을까 합니다.

그리고 하나의 AS가 서로 다른 AD들과 연결되어 있습니다. 예를 들면 "AS-03. 외부 권한 조회 다층 캐시 적용" 이 "AD-01. 피크 시 권한 갱신 응답시간"과 "AD-04. 핵심 기능 성공률 99.9%"와 연결되어 있습니다. 하나의 원인으로 발생하는 증상들을 각각 이슈로 보고 접근할 때 발생하는 문제일 것으로 보입니다. 처음에는 그럴 수 있지만, 분석을 통해 하나의 이슈로 정리를 해야 하는 상황이 아닐까 싶습니다.

한번 더 이슈를 잘 정리해서 AD를 뽑아 내고 AS를 정리해 보시는 것도 고려 해 보세요.

AS-02 에서 8만개의 동시 입장 처리를 비동기 처리를 함으로써 해결하려고 하시는데, thread 응답 대기 외 다른 자원의 문제는 없나요?  즉 서버를 늘리지 않아도 해결이 될 수 있는지 해서요. 또는 특정 외부 서버는 동시에 8만개의 동시 입장으로 인한 부하는 괜찮을까 싶습니다.

채택된 대안에는 한계 부분이 없습니다. 통상적으로 아키텍처는 Trade-off를 고려해서 그 중에서 나은 선택을 하는 것으로 알고 있습니다. 즉 어떤 대안이던 않좋은 부분이 있기 마련인데. 채택하신 대안에는 한계가 없습니다. 한 두개는 그럴 수 있는데 전부 다 그런다면 관련해서 질문이 나올 거 같습니다.

이슈-설계목표-기능요구사항-품질요구사항-드라이버에 대한 추적성은 시간 관계상 리뷰하지는 않았습니다. 양해 부탁드립니다. 이 부분은 한 번 더 잘 점검해 보시기 바랍니다.

- 🔴 **ISSUE-07이 진짜 아키텍처 이슈인지 재확인 필요** — "MariaDB에서 read/write lock 경합이 DB isolation 설정만으로 해결 안 되는가?"라는 실질적 기술 질문. Primary-Replica 분리(AS-07)로 가기 전에, 격리 수준 조정만으로 해소 가능한 범위인지 먼저 검토하고, 안 되는 이유를 이슈 본문에 명시하는 게 안전.

### report/2/2.1-기능요구사항.md (Use Case)

- 🔴 **extend 화살표 방향 오류 가능성** (L108-109) — `UC03 -.->|"«extend»"| UC08` 형태. UML 관례상 extend는 "확장 유스케이스 → 기본 유스케이스" 방향이므로 `UC08 -.->|"«extend»"| UC03`가 맞을 가능성 높음. 확인 후 수정 권장.

### report/3/3.2-아키텍처문제분석및설계전략.md

- 🟡 **AD/AS ID 순서 정리** — ISSUE ID 순서를 따라가다 보니 AD·AS 번호가 등장 순서와 안 맞는 구간 있음(표44). ID 부여 기준(우선순위 vs 이슈 발견 순)을 명확히 하거나, 가능한 범위에서 순서를 맞추는 것 검토. 평가자도 "임의 부여 자체는 괜찮다고 보나 평가자 기준은 다를 수 있다"고 함 — 우선순위 낮음.
- 🔴 **동일 AD를 공유하는 이슈 쌍들 — 이슈/드라이버 재정리 검토** (표44 기준):
  - ISSUE-01·03 → 둘 다 AD-02
  - ISSUE-04·07 → 둘 다 AD-03
  - ISSUE-05·06 → 둘 다 AD-04
  - ISSUE-02·09 → 둘 다 AD-01 포함

  평가자 지적대로, 하나의 근본 원인이 증상별로 이슈가 쪼개진 것인지, 정말 별개 이슈인지 재검토 필요. 합칠 수 있는 건 합치거나, AD를 더 세분화해서 1:1 관계로 정리하는 방향 고려.
- 🔴 **하나의 AS가 여러 AD에 연결되는 패턴** — AS-03(다층 캐시)이 AD-01·AD-04에 걸쳐 있는 것 외에도, AS-02(AD-02·AD-04), AS-08(AD-02·AD-03·AD-04), AS-09(AD-01·AD-04) 등 다수 존재. 위 이슈 재정리와 함께 다뤄야 근본적으로 해소됨.
- 🔴 **AS-02(비동기 전환)의 한계 미기술** — 스레드 대기 문제 외에 다른 자원(서버 증설 없이 해결 가능한지, Meeting Manager 자체가 8만 동시 요청을 감당할 수 있는지)에 대한 언급 없음. AS-02 절에 한계/전제 조건 추가.
- 🔴 **채택 대안에 한계(Trade-off) 서술이 전무** — AS-01~09 전체에서 "채택 대안"에는 단점이 하나도 안 적혀 있음(대안1,2에는 단점 명시, 채택안은 장점만). 아키텍처 결정은 트레이드오프이므로, 채택안에도 최소 1개씩 한계/제약을 명시해야 함. 가장 눈에 띄는 구조적 공백이라 우선순위 높게 권장.

> 이전 버전의 "5개"는 기준 없이 임의로 추린 목록이었음. 🔴 16개 항목을 다시 세어, 실제로 몇 개의 **근본 원인**이 여러 문서에 반복 노출된 것인지 기준으로 재그룹핑함. 아래 P0~P4에 16개 전부 빠짐없이 배치.
