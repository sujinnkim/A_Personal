# 4.1 제출 피드백 — P3 수정 내역

> 대상: [4.1 제출 피드백 및 수정포인트.md]의 **P3 (ISSUE-07이 진짜 아키텍처 이슈인지 사실 확인)**
> 평가자 지적: "MariaDB에서 read/write lock 경합이 발생하는가? DB isolation 설정으로 해결이 되지 않나?"
> 적용일: 2026-07-05 · docs·report 동시 반영

## 1. 사실 확인 결과 (평가자 지적 부분적으로 타당)

| 항목 | 확인 결과 |
| ----- | ----- |
| 순수 read/write lock 경합 | InnoDB 기본 격리(REPEATABLE READ)에서 일반 SELECT는 **MVCC consistent read라 공유 lock 미획득** → 순수 lock 경합은 격리 수준 조정(READ COMMITTED 등)으로 상당 부분 완화됨. **평가자 지적 일리 있음.** 기존 "read/write lock을 경쟁"이라는 서술은 엄밀히 부정확 |
| 실제 병목 (격리로 해소 안 됨) | (1) **write-write 경합**: 동일 레코드·인덱스에 INSERT(셀프)+UPDATE(cPaaS) 집중 — write는 격리수준 무관 항상 lock. (2) **Primary 단일 노드 write 처리량 포화**: redo log flush·디스크 I/O·버퍼풀 경합 — 격리수준 완전 무관. (3) write 중 read latency 상승 — 동일 노드 자원 경합 |
| AS-07(CQRS 방향) 정당성 | **유지 타당.** 근본 해법은 read를 Replica로 물리 분리해 Primary가 write에 집중하게 하는 것. 격리 조정으로는 (1)(2) 미해소 |

## 2. 이번에 반영한 것

**(1) 콜아웃 추가** (report 1.2 + docs ISSUE-07): "격리 수준 조정만으로 해소되지 않는 이유" — MVCC로 순수 lock 경합은 완화되나 write-write 경합·Primary 처리량 포화는 격리 무관하게 남으며, 근본 해법이 read/Replica 물리 분리(AS-07)임을 명시.

**(2) 전면 재프레이밍 (사용자 승인 후 진행)** — 부정확한 "read/write lock 경합" 병목 명칭을 실체("write 집중 경합 + Primary 처리량 포화 + read 자원 경합")로 전면 교체. 교체 표현 통일:
- 이슈 제목/이슈명: "참석자 상태 DB lock 경합" → "참석자 상태 write 집중 경합"
- 복합 원인 나열: "DB lock 경합" → "참석자 상태 write 집중"
- 상세: "read/write lock을 경쟁" → "write 간 경합 + Primary 처리량 포화 + read·write 노드 자원 경합"
- 다이어그램(그림7) 레이블: "⚠ read/write lock 경합" → "⚠ write 집중 경합·처리량 포화"

**반영 파일 (14개, docs·report)**: report 1.2·1.3·2.1·2.2·3.1·3.2 / docs ISSUE-07·1.3·UC-05·QA-02·QA-04·3.1·AS-07·4.1.1-concept-diagram. 이슈 제목 변경에 따른 참조처(QA-02 관련이슈, 3.2 표44, 저해요인 표 등)도 동기화.

**의도적 미변경 2건**: ① 콜아웃 내부의 "순수 read/write lock 경합" (설명 목적), ② `docs/01-plan/additional-info.md` (사용자 제공 참고자료, as-is 원본 기록).
