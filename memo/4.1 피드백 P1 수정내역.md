## 1. 원칙과 범위 판정

문제 정의 계층(이슈·요구사항·제약)에서 "무엇을 할지(해법)"를 빼고 "무엇이 문제인지(현상·원인)"만 남긴다. 단, **as-is 사실 기술은 해법이 아니므로 유지**한다(예: 현행이 allOf 대기라는 서술). 아래 4곳만 실제 해법 전제였고, 나머지 CR·QA는 전수 확인 결과 중립이라 미변경.

| 위치 | 해법 전제 (변경 전) | 성격 |
| ----- | ----- | ----- |
| ISSUE-02 제목·본문 | "서버사이드 캐시 부재", "캐시 없이" | 캐시(AS-03)를 문제 이름에 노출 |
| QA-04 응답 | "실패를 graceful하게 처리하여" | fallback(AS-09)을 품질기준에 노출 |
| CR-02 | "기능별 커넥션 풀 분리(Bulkhead)" | 격벽(AS-08)을 제약에 노출 |
| CR-07 | "로컬 캐시", "TTL 정책과 캐시 무효화 전략" | 캐시(AS-03)를 제약에 노출 |

## 3. 변경하지 않은 것 (과잉 방지)

- **CR-01·03·04·05·06**: 전수 확인 결과 이미 solution-neutral (동기 블로킹 특성, 점진 적용, API 호환성 등 사실·제약만). 미변경.
- **QA-01·02 "현행 문제" 서술**: "allOf 대기", "Feign 동기 호출" 등은 as-is 사실 기술이지 해법 전제가 아님. 유지.
- **3.2 AS 계층**: Bulkhead·Caffeine·Redis 등 해법 표현은 여기 있는 게 정상. 그대로 유지.

## 4. 검증

- 문제정의 계층(1.2·2.2·2.3)에 해법 표현(Bulkhead·Redis·Caffeine·TTL·graceful·캐시부재) **0건** 확인.
- 해법 계층(3.2 AS)은 해당 표현 유지 확인.
- 옛 ISSUE-02 제목 잔존 0건.
