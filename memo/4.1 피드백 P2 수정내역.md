# 4.1 제출 피드백 — P2 수정 내역

> 대상: [4.1 제출 피드백 및 수정포인트.md]의 **P2 (채택 대안의 트레이드오프 서술 공백)**
> 평가자 지적: "채택하신 대안에는 한계가 없습니다. 아키텍처는 Trade-off를 고려한 선택인데 채택안에 단점이 하나도 없다."
> 적용일: 2026-07-05 · docs(원본)·report(파생) 동시 반영

## 1. 원칙

기존 문서는 대안 1·2에는 한계를 적고 채택안(대안 3 또는 2)에는 장점만 적어, "채택안엔 단점이 없다"는 인상을 줬다. 아키텍처 결정은 트레이드오프이므로 **채택안에도 실제 한계를 명시하고, 그 한계를 어떻게 완화·감수하는지**까지 서술한다.

## 2. 반영 형식 (일관)

- **report 3.2**: ① 대안 비교표 채택행의 "—" 셀을 실제 한계 요약으로 채움, ② 채택 근거 문단 뒤에 "**채택안의 한계와 완화**" 문단 추가(한계 + 완화·감수 방식).
- **docs 3.2 (AS-01~09)**: 채택 근거 문단 뒤에 동일한 "**채택안의 한계와 완화**" 문단 추가.

## 3. AS별 채택안 한계·완화 요지

| AS | 채택안 | 핵심 한계 | 완화 |
| :--: | ----- | ----- | ----- |
| AS-01 | 선별적 도메인 모듈 분리 | ArchUnit 규칙 지속 관리, 같은 JVM 공유로 물리적 격리 제한 | CI 자동 검증, 런타임 격리는 AS-08이 보완 |
| AS-02 | @Async 하이브리드 | externalCallExecutor 신규 고갈점, 컨텍스트 전파·예외 복잡도 | TaskDecorator 컨텍스트 복사, AS-09 CB로 조기 차단 |
| AS-03 | L1+L2 캐시 | TTL 구간 stale(권한 반영 지연), Redis 운영·무효화 복잡 | 변경 빈도 낮은 데이터 한정, @CacheEvict 즉시 무효화 |
| AS-04 | 전용 Connector | 포트 라우팅 인프라 설정, 정적 배분 유휴/포화, 총 스레드 증가 | 라우팅 1회성, maxThreads 프로파일 기반 설정·모니터링 |
| AS-05 | 피크 예측 차등 스로틀링 | 돌발 급증 미작동, 429 UX 저하, 임계값 튜닝 | AS-04가 핵심 입장 보호(안전망), 429는 비핵심 한정 |
| AS-06 | 예약 기반 선제 초기화 | 예측 빗나가면 워밍 낭비·외부 부하, 스케줄러 조용한 실패 | 임계값(500명↑) 한정, 배치·딜레이 분산, 실패해도 AS-03 정상 |
| AS-07 | Primary/Replica 라우팅 | 복제 지연 read-after-write 불일치, readOnly 누락 위험 | 재조회 경로 Primary 유지, 강정합 불필요 조회만 Replica |
| AS-08 | 이중 Bulkhead | 정적 분할로 총량 증가(CR-02 압박), 배분 오류 시 특정 풀 고갈 | 풀 합 DB 상한 내 설계, 풀별 메트릭 모니터링 |
| AS-09 | 차등 CB + 계층 Fallback | 서버별 CB 튜닝 부담, Fallback 데이터 stale, 상태 관리 복잡 | 특성 기반 초기값(표56) 제시, Fallback은 저정합 데이터 한정 |
